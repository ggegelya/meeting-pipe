---
description: Multi-angle repo assessment - parallel finder passes, adversarial verification, owner-gated filing
---

You're running an owner-commissioned audit. Scope: $ARGUMENTS (when empty: the whole repo since the last audit band; the band-origin sections in the active backlog are dated, and the last full one was the assessment review 2026-07-12).

This codifies that assessment's method. It is expensive by design (many parallel agents) and runs only when the owner invokes it. Before starting, note two practical things to the owner: long multi-agent runs die under clamshell sleep, so the Mac stays lid-open or under `caffeinate -i` for the duration; and a full run takes a while and burns real tokens, so confirm the scope before fanning out.

0. Sweep stale worktrees first: `sh scripts/prune-worktrees.sh`. It removes a worktree under `.claude/worktrees/` only when every gate clears: merged into `main`, at least one commit made, idle, no live session transcript, clean tree, and never the one you are standing in. A session that has not committed yet is always kept, which is the fix for the 2026-07-24 incident where a live session lost its worktree. Silent no-op when nothing is stale. A `SessionStart` hook runs it too; this line is the backstop for when the hook isn't installed. If you ever change that script, `sh scripts/test-prune-worktrees.sh` is mandatory.
1. Read the orientation set first: root `CLAUDE.md`, `ARCHITECTURE.md`, and the active backlog's ToC plus band origins, so findings that are already filed tasks get recognized rather than re-filed.
2. Pick 4-8 finder angles fitting the scope, from the proven menu: daemon architecture; pipeline architecture + cross-language contracts; SOLID/KISS/YAGNI; doc/functional consistency (docs vs code truth); coverage + test quality (actually run both suites and read what they skip); detection stability; UX/setup; product opportunities. A scoped audit picks only the angles that can see the named subsystem.
3. Run the finders as parallel read-only subagents (the Agent tool; for a full-repo run, the Workflow tool with verification pipelined behind finding). Each finder returns structured findings: the claim, `file:line` evidence, a proposed severity (critical / high / medium / low), and what it could NOT see. "No signal" is not "not broken": unrunnable suites, missing hardware, and permission-gated paths get named in the report, never silently skipped.
4. Adversarial verification: every critical and high finding goes to an independent skeptic agent instructed to REFUTE it in code, not to confirm it. Only findings that survive keep their severity; downgrades and refutations are recorded and reported (the 2026-07-12 precedent: verifiers downgraded five of eleven highs and refuted one absolute, and the corrected severities are what got filed). Spot-check mediums; lows ride on the finder's evidence.
5. Synthesis: dedupe across angles, drop anything already filed (link the existing ID instead), band the survivors per the backlog's P1/P2/P3 definitions, and present the full corrected list to the owner: finding, evidence, severity, proposed band, proposed task shape. Nothing is filed before the owner approves the list (approve all, some, or re-band).
6. File the approved items in the `/file-task` format: one ToC row plus one `docs/backlog/tasks/<ID>.md` per item, and one band-origin paragraph in the backlog's Task specs section recording the date, the method (angles run, verification stats, what was refuted or downgraded), and the owner's approval date. One commit on `main`: `chore(backlog): file the <YYYY-MM-DD> audit band`. Don't push. No em-dashes; single-line paragraphs.
7. **Land the band if this session is running in a worktree.** Detect it first: inside a linked worktree `git rev-parse --path-format=absolute --git-dir` and `git rev-parse --path-format=absolute --git-common-dir` return different paths (`<repo>/.git/worktrees/<name>` against `<repo>/.git`); in the main checkout they return the same path. If they match, step 6 already committed on `main` and there is nothing to land. If they differ:
   - Resolve what you need, the main checkout and your audit branch: `MAIN=$(git worktree list --porcelain | head -1 | cut -d' ' -f2-)` (the first entry is always the main checkout) and `BRANCH=$(git rev-parse --abbrev-ref HEAD)`.
   - Preflight one thing only: the main checkout must be on `main` (`git -C "$MAIN" rev-parse --abbrev-ref HEAD`), or the merge below lands on whatever branch it happens to have checked out. A dirty main checkout does not block you; a fast-forward leaves unrelated uncommitted work alone, and git refuses on its own when the incoming commit touches a file the owner has modified there. Never stash, reset, checkout, or commit work you did not make in order to clear the path.
   - Rebase so the landing stays linear: `git rebase main`. An audit files several rows into the same ToC table a parallel session may also be appending to, so conflicts are expected; keep both sides' rows rather than picking one, and don't reorder rows you didn't add.
   - Re-check every ID you filed after any rebase that moved your commit. A parallel `/file-task` can take a number you claimed, and the band that lands second renumbers: re-grep `docs/backlog/*.md` and the whole repo per ID, and for each collision take the next free number and update the ToC row, the `docs/backlog/tasks/<ID>.md` filename, its `# <ID>:` heading, and any cross-reference in the band-origin paragraph together before landing.
   - Land it: `git -C "$MAIN" merge --ff-only "$BRANCH"`. `--ff-only` is deliberate: it keeps the band as one linear commit, and it refuses rather than papering over a base you forgot to rebase onto. If it fails because the owner's uncommitted work is in the way, stop and name the files it listed. If it fails on `index.lock`, another session is mid-merge, so wait a few seconds and retry once, then stop and report.
   - Clean up: if this session created the worktree itself, exit it with the remove action. If the worktree came from the new-conversation checkbox instead, that tool is a documented no-op (it only touches worktrees it created), and you must not delete the directory you're standing in either: doing so works, but every later tool call in this session then dies on a missing cwd, so a follow-up question would find a broken session. Leave it, and say it is already merged and that `scripts/prune-worktrees.sh` takes it at the next session start. To land it sooner, print both halves, since the worktree and the branch are separate: `git -C "$MAIN" worktree remove <path>` and `git -C "$MAIN" branch -d "$BRANCH"`.
   - Don't push, in either mode. Say in the summary that you landed from a worktree, and call out any renumber the rebase forced.
8. Findings without evidence do not get filed, at any severity. An angle that found nothing reports that it found nothing and what it looked at; silence is not clearance.
