# meeting-pipe Codebase Audit

Produced 2026-05-22 by a parallel multi-agent code audit: eight focused passes covering Swift core architecture, the MeetingPipeCore verdict-fusion engine, the audio subsystem, the SwiftUI layer, the Python pipeline, security, cross-cutting contracts and documentation drift, and UX/product. Findings were then cross-checked against the Q2 backlog and the owner's account of lived pain points. Scope at audit time: roughly 35,400 Swift LOC, 8,700 Python LOC, 218 commits.

## Verdict

This is a well-built project, not a rescue case. There is real engineering discipline: ADRs, a detailed backlog, around 500 tests, careful refactors. The problems are specific and tractable.

The original audit's headline concerns were: documentation that described a deleted architecture (resolved in Step 1); the owner's primary use case, Google Meet run as a PWA, sitting on the weakest detection path; a recording-capture layer with real-time-safety gaps; and a privacy guarantee that is documented but not enforced.

One framing correction, made after cross-checking the backlog: the MeetingPipeCore verdict-fusion subsystem is a deliberate, documented architecture (the TECH-C13 and TECH-G-MIC specs, ADR 0008), not accidental over-engineering. An earlier draft recommended tearing it out; that is withdrawn. The real issue is uneven execution. The engine is rich for Teams, where detection works, and thin for Google Meet, where it does not. The fix is to finish and rebalance it, not to remove it.

## 1. Documentation integrity (Resolved in Step 1)

ASR and diarization moved out of Python into the Swift daemon (FluidAudio / Parakeet), and the old Detector was replaced by the MeetingPipeCore lifecycle and MicGate subsystems. None of the docs followed. SPEC.md, ARCHITECTURE.md, README.md, CONVENTIONS.md, GLOSSARY.md, both CLAUDE.md files, and docs/test-coverage.md all described deleted modules and a deleted streaming pipeline. For a vibecoded project this was the highest-leverage problem: every agent reads ARCHITECTURE.md before non-trivial work, so every session started from a wrong map.

Step 1 refreshed all eight files. They now describe the FluidAudio plus MeetingPipeCore reality. Future sessions can trust them.

## 2. Meeting detection: Google Meet / PWA (Resolved in Step 2)

This is the owner's main lived pain and the audit's biggest correction. Step 2 closed both gaps below in commits `458986d` and `e8bf8a0`; the original findings are kept for context.

- [Critical] Google Meet runs on a single fragile signal. TECH-C13 itself calls Meet "the documented hardest case." Teams end-detection fuses three primary signals; the browser path (`BrowserMeetingLifecycleAdapter`) has exactly one, the tab title via a 2 Hz semaphore-blocked `ShareableContentSignal`. That asymmetry is why detection works only in Teams.
- [Critical] Google Meet run as a PWA is not detected at all. A Chrome-installed PWA runs under a synthetic bundle id like `com.google.Chrome.app.<hash>`, which is in neither bundle list in `daemon/Sources/MeetingPipe/Resources/meeting_apps.toml`, so `MeetingSourceScanner` never builds a candidate. Documented as backlog item TECH-I5 (an unscheduled idea, dated 2026-05-21).
- [High] The fix is partly already built. TECH-C13 specified four corroborating signals (`WindowTitleSignal`, `WorkspaceSignal`, `InputDeviceSignal`, `CalendarContextSignal`, under `Sources/MeetingPipeCore/Lifecycle/Signals/`). They were built but never wired into `PromotionEngine`. Wiring WindowTitle and Workspace into the browser adapter is the cheapest way to give Meet more than one signal.
- [Medium] The detection regression corpus (TECH-C6) is synthetic-only; the 20-plus real traces are owed and would catch Meet regressions.

## 3. Capture integrity: recording audio (Resolved in Step 3)

- [Critical] Blocking I/O on the capture callbacks. The mic `installTap` closure and the `SystemAudioCapture` callback call `AVAudioFile.write(from:)` synchronously on the AVAudioEngine tap queue and the SCStream delivery queue (`MeetingRecorder.swift`, lines ~247 and ~302). Those are latency-sensitive delivery threads; a blocking disk write under disk pressure can drop buffers. The fix is to move the write off the callback behind a preallocated ring buffer drained by a writer queue. Worth doing as real-time-safety hygiene regardless. Validation-pass correction: this was originally called the most likely root cause of the mic-ended-earlier skew, which is unconfirmed. The code itself offers a competing hypothesis (the `stop()` comment near line 411 blames the ffmpeg merge step), and teardown ordering is ruled out (`stop()` halts the system stream before removing the mic tap, which would make mic longer, not shorter). The skew (open P1 backlog bug, "End-of-call timeline shift") is a separate `/investigate` task.
- [High] No mid-recording device-change handling. Nothing observes `AVAudioEngineConfigurationChange`; unplugging AirPods mid-meeting silently flatlines the mic channel for the rest of the call.
- [Medium, corrected] The recording spine runs "main queue by convention," not `@MainActor`. An earlier draft called `Coordinator.latestSystemLevelDb` a live data race; the validation pass found the level callbacks are already marshalled to main (`MeetingRecorder.accumulateAndEmit` ends each emit in `DispatchQueue.main.async`), so the field is written and read on main. `@MainActor` on the spine would still help by turning the convention into a compiler-checked invariant, but it is hardening, not a bug fix, and can cascade across the ~1,600-line Coordinator. Scope it small or defer.
- [Medium] Verify "join muted." Confirm `AXMuteButtonProbe` emits the initial muted state at engage, not only on toggle, so joining a Teams call already muted gates the mic from frame one. The backlog P0 runtime check (no spurious `muted_by_app` drops to RMS) is still owed.

## 4. Observability and failure visibility (High, mostly resolved)

- [Critical, RESOLVED in Step 1] `mp analyze-detection` reported false results. It paired on a `detector.ended` event that TECH-C13 renamed to `lifecycle.ended`, so it classified every session as a detection miss. Fixed in `e7aa537`.
- [High, RESOLVED in Step 5] Pipeline failures vanish. A failed transcribe, summarize, or publish was one transient notification banner, silenced under Focus modes, with no persisted reason and no aggregate indicator. The owner could silently lose meetings. Step 5 added the durable `<stem>.error.json` sidecar, the Library `.failed` row state with the reason, and the aggregate status-bar row. Caveat: a partial-publish failure is still invisible (the pipeline's `publish_router` swallows per-sink failures by design); that sub-case is deferred.
- [High, Open] No in-app pipeline progress. Progress is menu-bar text only; a wedged subprocess is indistinguishable from a slow one without tailing logs.
- [Medium, Open] `dogfood-report.swift` (TECH-E4) is unbuilt, so the dogfood bars cannot be measured yet.

## 5. Security and privacy (High, Open)

- [High] Regulated-mode "zero egress" is not enforced. `regulated_mode = true` only suppresses the Notion publish. With the default `backend = "anthropic"`, a confidential transcript is still sent to Anthropic. The per-meeting NDA path already forces local; apply the same to the global flag.
- [Medium] The local LLM bind host is user-controllable. A stray `0.0.0.0` in config exposes an unauthenticated inference server, and the transcripts sent to it, on the LAN.
- [Medium] `secrets.env` permissions are enforced on write but not validated on read.
- [Low] `.gitignore` excludes `*.wav` but not `*.summary.md` / `*.meta.json`; transcripts could be committed if the output dir ever sits in a git tree.

Baseline posture is otherwise solid: no shell-injection surface, TLS never disabled, no secret leakage into logs.

## 6. Correctness gaps (High, partly addressed by owner)

- [High, Open] `ObsidianPublisher` idempotency is broken. The body hash includes a `generated` timestamp, so every republish rewrites the file, and a next-day re-run creates an orphan note.
- [High, partly addressed] Pipeline crashes orphan recordings. A failed `mp run-all` left a WAV plus transcript with no summary and no automatic retry. The owner has committed orphaned-recording recovery (`5577f3b`); re-verify what that covers and whether a Retry affordance is still wanted.
- [Medium, Open] `LocalSummaryClient` leaks its `mlx_lm.server` subprocess in the plain `local` backend path (the `auto` path scopes it correctly).

## 7. Architecture, structure and dead code (Medium, Open)

- [Medium] TECH-H1 is marked done but incomplete. Its acceptance was "Coordinator under 400 lines"; it is around 1,600. The subordinate types were extracted; the slimming was not finished. A `MeetingLibraryService` extraction is the natural next cut.
- [Medium] Untyped `[String: Any]` summary data is plumbed through four SwiftUI views, with file I/O inside view code. A typed `MeetingSummary` model removes the silent-failure stringly-typed access.
- [Medium] God views: `PreferencesView.swift` (around 1,100 lines), `MeetingDetailView.swift` (around 930).
- [Medium, mostly resolved in Step 4] MeetingPipeCore is deliberate architecture that needed finishing and tightening, not removal. Step 4 collapsed the near-duplicate per-app adapters to data (10 classes to 3) and corrected the bus listener-sharing docs. Still open: `InputDeviceSignal` and `CalendarContextSignal` are built but unwired (Step 2 wired only `WindowTitleSignal` and `WorkspaceSignal`) and should be wired or removed. Correction: `rmsCloseDwellMillis` is NOT dead, contrary to an earlier draft of this audit. The Step 4 session found it is read by `MicGate.decide`, emitted as the `dwell_ms` event attribute, and pinned by tests; it is vestigial (never populated past 0) but live, so removing it is an event-schema change, not a dead-code sweep.
- [Medium] DRY: three button systems, four hex-to-Color copies, three near-identical subprocess blocks in `PipelineLauncher`.
- [Low, resolved in Step 4] Dead code: the pyannote / HuggingFace block in `endpoints.py` and the stale `transcribe*.pyc` artifacts were removed.

## 8. Performance, rendering and UX polish (Medium / Low, Open)

- [Medium] `MeetingStore.scan` re-parses every sidecar in the library on a 500 ms debounce during recording; no mtime cache.
- [Medium] The onboarding screen is designed but unbuilt. `design/ui_kits/macos_app/OnboardingPermissions.jsx` exists; the real first run fires four unframed TCC dialogs.
- [High, within UX] Recovery actions (Retry, Regenerate, Republish) are right-click-only, and the long-meeting and BYO flows dead-end at a terminal command.
- [Medium] The UI addendum's 10 polish tasks are unbuilt, and were written against a `Sources/MeetingPipeLibrary/` layout that does not exist; their paths need correcting before delegation.
- [Low] Render cliffs: `WaveformBody` redraws the full Canvas at 15 Hz; `FluidAudioRunner` loads the whole recording into RAM.

## What is already good (do not regress)

The two-target Swift split (MeetingPipe app, MeetingPipeCore library); clean Swift memory management; above-average SwiftUI performance discipline (revision fingerprints, a split processing tracker, TimelineView animations, deduped permission polls); the well-reasoned Notion publisher; the test suite's use of real in-memory fakes; no shell-injection surface; `.gitignore` covering secrets and recordings. The playback mono-mixdown the owner asked for is done (TECH-LIB-MIX, `PlaybackChannelMode.swift`); the left-ear / right-ear problem is solved.
