# meeting-pipe — what's left + next direction

Companion to `meetingpipe-audit.md`. Steps 1-5 of the audit are done and verified. This file captures what was not closed, ideas worth considering (yours and mine), and a suggested path.

## What's not done from the audit

### The one open P1 — your actual lived pain
- **Mic / system end-of-call skew.** Step 3 fixed the underlying real-time-safety hazards, but the skew itself was deferred to a separate investigation. Two candidate root causes still on the table: capture-callback buffer drops, or the ffmpeg merge step at the end of recording.

### Owner-side runtime acceptance owed (only you can do these)
- Live Meet PWA detection in a real call.
- Live device swap mid-recording (unplug AirPods, confirm one continuous file).
- The three new Step 5 failure surfaces (inline Retry button, detail-pane failed state, status-bar "N failed" row).
- The join-muted check (confirm mute state is captured at meeting start, not only when you toggle).

### Quality items still open (no urgent user pain)
- **Coordinator slimming** — still about 4× over the original target size. A library-service extraction is the natural cut.
- **Typed summary model** — untyped dictionary data is threaded through four detail views, so a typo silently shows an empty section.
- **God views** — the preferences and meeting-detail views are both over 900 lines.
- **Design-system drift** — three button systems, four copies of color-from-hex, three near-identical subprocess wrappers.
- **Library scan has no mtime cache** — re-parses every sidecar on a 500 ms debounce while recording.
- **Two corroborating detection signals are built but never used** (input-device, calendar-context). Wire them or remove them.
- **Render cliffs** — waveform redraws the full canvas at 15 Hz during playback; the transcription loader keeps the whole recording in memory.

### UX gaps the audit calls out and Step 5 only partly addressed
- **Onboarding is unbuilt.** A mockup exists; today first run fires four unframed permission dialogs in a burst — dismiss one and the rest are silently dead.
- **Regenerate / Republish are still right-click-only.** Step 5 surfaced Retry; the other two recovery actions did not get the same treatment.
- **Long-meeting and BYO flows dead-end at a terminal command** — no in-app completion path.
- **Degraded recording state surfaces only after the meeting** — when system audio capture failed at start, you find out too late.
- **No in-app pipeline progress** — a wedged subprocess looks identical to a slow one without tailing log files.
- **Ten UI polish tasks from an earlier addendum** — paths need correcting before they can be delegated.

### Smaller residuals tracked through the audit
- Two pre-existing telemetry quirks in the mute-gating adapters (browser-vs-meet shadowing; PWA bundle IDs match no adapter). Telemetry-only, safe to defer.
- Secrets file permissions are validated on write but not on read.
- Partial publish failures stay daemon-invisible (a Notion outage with a successful local Obsidian copy is not surfaced).

## Your ideas

1. **Drop Python entirely.** Feasible but premature. The pipeline is stable and well-tested; the dependency is a one-time setup. Real value comes only after idea 2 lands and removes most of why Python is there.

2. **Apple Intelligence for summarization.** Strongest near-term direction. macOS 26 ships an on-device Foundation Model — Swift-native, free, no setup, no API key, no network. Catch: 4K context limit, so long meetings need chunked summarization. Add as a coexisting backend; make it default once it proves out.

3. **Local LLM improvements + wire the Corrections tab.** First step is to actually check whether the Corrections tab is wired end-to-end today — corrections are persisted, but the UI loop may not be complete. Local-LLM improvements (prompt tuning, model-size in the UI, faster cold-start) are real but each individually small.

4. **Live transcription during recording.** Biggest feature on your list. The project previously had streaming and deleted it; bringing it back reverses a documented decision. The right path today is Apple's macOS 26 SpeechAnalyzer (on-device, streaming, Apple-supported). Real cost: extra CPU while recording; must not nudge mute-detection accuracy. Eventual, not next.

5. **LLM-based diarization cleanup.** Highest leverage per hour on your list. After the diarizer assigns speaker labels, an LLM post-pass merges clearly-same-speaker chunks and re-attributes obvious mistakes (e.g. when one speaker says "thanks Tom", the next utterance is probably from Tom). Adds a few seconds to summarize, visibly cleans transcripts. Live diarization improvement is fundamentally harder and not realistic.

6. **HUD icon for the currently-recording app.** Easy. The detection layer already knows which app it is; the HUD just needs the icon slot. Half-day session.

7. **First-launch onboarding.** Already in the audit (mockup exists). Real UX pain today. Medium-sized session.

8. **Opt-out auto-restart on quit.** Easy. Preferences toggle plus clearer Quit menu copy, optionally a "Quit (don't relaunch)" alternate item. Default OFF is defensible since you are the only user.

## My added suggestions

- **Surface degraded recording state on the HUD during the meeting**, not after. Currently half-lost data is discovered too late.
- **Voice-activity meter on the HUD** — small UX win, pairs naturally with the HUD app icon.
- **Regenerate / Republish discoverability** — same treatment Retry got in Step 5.
- **Long-meeting / BYO in-app completion** — first-class entry points deserve first-class completion paths.

## Suggested path

Pick one lane for the next session.

**Lane 1 — Close the open P1.** Investigate the mic/system end-of-call skew. The one outstanding audit item that maps to your actual lived complaint, deferred through three steps now. Highest value to you personally.

**Lane 2 — Two quick UX wins, one session.** HUD app icon (your idea 6) + opt-out auto-restart (your idea 8). Both immediately visible next launch, both well-scoped, good momentum.

**Lane 3 — Strategic shift.** Apple Intelligence backend (your idea 2). Most important new direction because it unlocks dropping Python. One session for the backend wiring and chunking strategy; coexist with the existing two; prove it on real meetings.

After whichever you pick:
- **Lane 4** — LLM diarization cleanup (your idea 5). High leverage, low effort.
- **Lane 5** — Onboarding screen (your idea 7 + audit topic). Mockup exists.
- **Lane 6** — Failure-visibility follow-ups (Regenerate/Republish discoverability, degraded-recording HUD, in-app pipeline progress) as one focused session.
- Later — Quality items (Coordinator slim, typed summary, god views, design-system) at the project's natural cadence.

**Avoid for now:** drop Python entirely (premature without Lane 3 proving out); live transcription (biggest feature, reverses a documented decision, needs Lane 3's groundwork).
