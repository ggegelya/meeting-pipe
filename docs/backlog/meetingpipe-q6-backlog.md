## How to pick up a task (delegation)

Mechanics are codified in `/tech-task <ID>` (read the task here, read the orientation docs, implement, verify, one commit on `main`, do not push). Soft context that command does not carry:

- Task ids carry no `TECH-` prefix (e.g. `/tech-task STOR1`). Historical code comments, ADRs, and commits keep the old `TECH-` form as provenance.
- **Full specs for items carried from Q4 unchanged live in [`q4-final.md`](./q4-final.md)** (search the id); the Task specs section below restates the new and changed items and points to q4-final for the rest, so each spec has a single owner.
- User locale: 99% English / Ukrainian. Do not default examples to German. Verify any new `uk` labels in `MuteLabels.toml` with the user.
- The user is vibe-coding: high-level explanations, does not read code. Lean on the ARCHITECTURE.md diagrams. Blind sub-pixel visual churn is out of bounds; visual changes need eyes, and the daemon is a menu-bar app that is not screenshotable here, so a rendered eyeball is owner-owed.
- No em-dashes in any output (CI fails on a U+2014 in an added line under `daemon/Sources` and `daemon/Resources`). Use hyphens, commas, or rewrite. Commit with the repo's git identity. Do not push without permission.
- Binding constraint (CLAUDE.md): the primary user is the author; selling is tertiary; technical excellence in core functionality comes first. The BRAND launch band stays deferred. The 2026-07 redesign band (DSN20-29 + WF6 + UX15) shipped in Q5; the only redesign residual is the owner-run `/design-sync` push, an outward sync tracked as an owner action, not a task.
- The AI/DV standing rule: every `AI*`/`DV*` capability and any new engine call must route through the local engine path so the `egress_guard` clamps it under regulated/NDA, honour `effective_backend()` (forced local under regulated/NDA), and add no new always-on egress.
- The de-CLI standing rule (2026-07-10): the CLI is a debugging escape hatch, not a product surface. A task that ships a user-facing capability includes its UI affordance in the same task; a CLI-only ship is `partial`, not done. Exempt by class (owner-dev diagnostics): dogfood, analyze-detection, corrections-stats, train-adapter, spike harnesses. Rationale: the Q4 validation audit carried 13 tasks for the CLI-only-no-UI pattern, it recurred in Q5 (AI4), and unwritten goals lose to written personas in an AI-steered repo.

## Marking a task done

When a task ships done-done, move it to the running quarter archive [`q6-final.md`](./q6-final.md) in the same commit: its ToC row (Status `done`, a one-line ship note in the Comment) and its full Task-spec text both move there, so the active backlog stays lean and the archive keeps the whole trail (process change 2026-07-10; before that, done rows sat here until the quarter roll). A partial ship stays here as `partial` with the remainder named in the Comment; owner-owed runtime or eyeball work stays here as `owner-owed`. `/tech-task` codifies the move.

**CONC4 (P2, partial): extend compiler-checked concurrency to the executable.** `-strict-concurrency=targeted` covers only MeetingPipeCore ("an island", per Package.swift); the 104-file executable mixes four idioms (47 `Task {`, 15 `Task.detached`, 67 DispatchQueue sites, 36 `@MainActor`, plus Combine) with comments as the only enforcement, and this exact bug class has shipped three incidents (the 2026-05-13 VPIO degradation, the 2026-06-05 stopCapture hang, the 2026-06-12 stacked starts). Acceptance: the executable builds warning-clean under targeted strict concurrency; no behavior change; the full suite stays green.

**Progress 2026-07-10 (measured, then stopped deliberately).** Flipping the flag produces **20 unique warnings** (the raw build prints ~400; they are the same 20 repeated per compilation unit). By file: `MeetingRecorder` 7, `CorrectionsTab` 5, `MeetingSessionController` 2, `SinkDispatcher` 2, `DoctorRunner` 2, plus 2 surfaced in Core.

The 2 Core ones **are shipped**: `MeetingLifecycleVerdict`, `MeetingLifecycleContext`, its nested `Kind`, and `EndingReason` are now `Sendable`. They are immutable value types crossing an `AsyncStream` boundary, exactly like `MicGateVerdict`, which already declared it; annotating them is correct on its own terms and is a prerequisite for the rest. The flag itself was reverted so `main` stays warning-clean.

The remaining 18 are **not** the "mechanical Sendable annotations, not logic changes" the spec anticipated, which is why this stopped rather than guessing:

- `MeetingSessionController` (2) and `SinkDispatcher` (2) capture `self` into `@Sendable` closures. Both are documented main-queue-only, so the correct fix is `@MainActor` on the type, not `@unchecked Sendable`. That is a real isolation decision: it cascades into `Coordinator` (which calls them from non-isolated context), into `SessionHost`'s protocol requirements, and into ARCH4's new tests.
- `MeetingRecorder` (7) captures `AVAudioEngine`, `SystemAudioCapture?`, and two optional callback closures into `@Sendable` closures, plus wants `@preconcurrency import AVFoundation`. Its cross-thread state is lock-guarded, so `@unchecked Sendable` is arguable, but it is a promise about synchronisation, and the audio-render-thread rules mean a wrong one silences a warning while leaving a real race.
