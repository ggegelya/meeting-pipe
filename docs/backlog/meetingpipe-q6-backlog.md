## How to pick up a task (delegation)

Mechanics are codified in `/tech-task <ID>` (read the task here, read the orientation docs, implement, verify, one commit on `main`, do not push). Soft context that command does not carry:

- Task ids carry no `TECH-` prefix (e.g. `/tech-task STOR1`). Historical code comments, ADRs, and commits keep the old `TECH-` form as provenance.
- **Full specs for items carried from Q4 unchanged live in [`q4-final.md`](./q4-final.md)** (search the id); the Task specs section below restates the new and changed items and points to q4-final for the rest, so each spec has a single owner.
- User locale: 99% English / Ukrainian. Do not default examples to German. Verify any new `uk` labels in `MuteLabels.toml` with the user.
- The user is vibe-coding: high-level explanations, does not read code. Lean on the ARCHITECTURE.md diagrams. Blind sub-pixel visual churn is out of bounds; visual changes need eyes, and the daemon is a menu-bar app that is not screenshotable here, so a rendered eyeball is owner-owed.
- No em-dashes in any output (CI fails on a U+2014 in an added line under `daemon/Sources` and `daemon/Resources`). Use hyphens, commas, or rewrite. Commit with the repo's git identity. Do not push without permission.
- Binding constraint (CLAUDE.md): the primary user is the author; selling is tertiary; technical excellence in core functionality comes first. The BRAND launch band stays deferred. The 2026-07 redesign band (DSN20-29 + WF6 + UX15) shipped in Q5; the only redesign residual is the owner-run `/design-sync` push, an outward sync tracked as an owner action, not a task.
- The AI/DV standing rule: every `AI*`/`DV*` capability and any new engine call must route through the local engine path so the `egress_guard` clamps it under regulated/NDA, honour `effective_backend()` (forced local under regulated/NDA), and add no new always-on egress.
- The de-CLI standing rule (2026-07-10): the CLI is a debugging escape hatch, not a product surface. A task that ships a user-facing capability includes its UI affordance in the same task; a CLI-only ship is `partial`, not done. Exempt by class (owner-dev diagnostics): dogfood, analyze-detection, corrections-stats, train-adapter, spike harnesses. Rationale: the Q4 validation audit carried 13 tasks for the CLI-only-no-UI pattern, it recurred in Q5 (AI4), and unwritten goals lose to written personas in an AI-steered repo.

## Marking a task done

When a task ships done-done, move it to the running quarter archive [`q6-final.md`](./q6-final.md) in the same commit: its ToC row (Status `done`, a one-line ship note in the Comment) and its full Task-spec text both move there, so the active backlog stays lean and the archive keeps the whole trail (process change 2026-07-10; before that, done rows sat here until the quarter roll). A partial ship stays here as `partial` with the remainder named in the Comment; owner-owed runtime or eyeball work stays here as `owner-owed`. `/tech-task` codifies the move.

**MIC14 (P2): off-the-record manual redaction spans.** Mute redaction only protects spans where the meeting app was muted; the human cases (a side conversation, an interruption, "this next part stays out") have no affordance. Scope: a hotkey toggles an off-the-record span mid-recording; spans are recorded into the existing mute-timeline machinery tagged `source: manual` (`<stem>.mute-timeline.json` grows a per-span source field, version-bumped and documented), and MuteRedactor redacts them in the transcribed, summarized, and published artifact while the full recording stays local for recovery, the ADR 0016 posture. Two invariants: a manual span is explicit intent, so it is always redacted and exempt from MIC12's speech-bearing withhold; and under the regulated/NDA real-time gate path the hotkey zero-fills live instead (no unredacted audio at rest there). The HUD shows a truthful persistent state while active ("Off the record") so it cannot be forgotten, and an open span auto-closes at stop. Acceptance: span audio is absent from transcript, summary, and every published artifact and present in the local original (capture-first path); the HUD state is truthful; the timeline round-trips; MIC12's guard leaves manual spans alone.

**MIC15 (P1): capture-time input-device identity plus a dead-mic warning.** Owner-reported 2026-07-07 from meeting `20260707-150057`: a 4m30s Teams-in-Safari call recorded the user's own voice as 4.5 minutes of noise floor (mic channel RMS -74.9 dB, peak -45.8 dB) while the system channel captured the far side cleanly, and nothing warned. Root cause, confirmed in the recorder: the daemon captures `AVAudioEngine.inputNode`, which macOS binds to the System Settings default input, and it neither reads nor can read the input device the meeting client selected, so a Bluetooth headset left as the system default (mic idle in A2DP) is recorded faithfully while the user speaks into a different mic. There is deliberately no device picker (config.example.toml), and there is no macOS API to read another app's chosen CoreAudio input, so following the meeting's device directly is out of scope; the fix is to record what we captured and to catch the mismatch. Scope, three layers: (a) at capture start, resolve the bound input device to a human-readable identity (name, uid, transport built-in/bluetooth/usb/aggregate, sample rate) via `kAudioObjectPropertyName` / `kAudioDevicePropertyDeviceNameCFString` and record it, to the event log (`recorder` category) always and persisted per-meeting so the Library detail can show which mic a recording used; if persisted into `<stem>.meta.json`, update the Swift writer, the CONVENTIONS schema, and the CI2 golden fixture together, since that file is the Swift-to-Python contract; (b) a post-stop backstop warning, symmetric to the existing `RemoteAudioWarning` (which only fires when system audio never arrived): when system audio was present for the whole call but the mic never crossed a plausible-speech floor across any non-muted span of a recording longer than ~60s, emit a Notifier alert ("Your mic recorded almost nothing. Check System Settings, Sound, Input") plus an event and a Library row flag, using MicGate's muted spans to exclude the legitimate muted-whole-meeting case; (c) a start-time mismatch nudge that reuses END6's `kAudioDevicePropertyDeviceIsRunningSomewhere` HAL read: if the device we are about to record is not the input device currently running-somewhere while another input device is, warn before the meeting is lost rather than after. Layer (b) is the must-have; (a) makes the class of bug diagnosable at all (today no device name is logged anywhere in the daemon); (c) is the early-catch upgrade. Acceptance: a recording whose mic stayed silent under an active system channel produces a visible warning and an event; every recording logs and shows its captured input device; the muted-whole-meeting case does not false-warn; no device picker is added.

**LOCAL9 (P2): corrections Phase 3, local LoRA fine-tune.** `mp corrections-stats` advertises a Phase 3 readiness check (~20 corrections over ~200 minutes) that no backlog task owns; make it real or stop advertising it. Scope: an owner-invoked `mp train-adapter` that fine-tunes a LoRA on the standard 7B via the MLX LoRA path, building prompt/completion pairs from the corrections corpus (original vs corrected summaries); evaluation is `mp dogfood` A/B (base vs adapter) on held-out meetings, hand-graded like the engine comparison; the adapter ships only through an explicit config opt-in (`summarization.local_adapter_path`), never silently applied. Fully on-device, no egress; training is a command the owner runs, no background training. An honest terminal outcome is allowed: if the A/B shows no capture win, close as adapter-not-worth-it, keep the corpus as a prompt-context asset (few-shot exemplars are the cheaper fallback lever), and fix the README wording. Gated on the corpus bar, which is owner-owed grading volume. Classification note (2026-07-10): `mp train-adapter` is owner-dev tooling under the de-CLI standing rule, CLI-only by design, like dogfood. Acceptance: with the bar met, a training run produces a loadable adapter; the dogfood scorecard decides adoption; README's corrections section describes the real flow either way.

**FEAT3-UNDO (P2, sequence ahead of FEAT3-SEGMENT): reversible speaker naming.** Owner-reported 2026-07-07. "Name this speaker..." (FEAT3-ROSTER) is a one-way door: `mp roster enroll` folds the speaker's embedding into the named person's roster centroid AND relabels every segment of that cluster in `<stem>.json` in place, so the name shows at once, with no in-app way back. The reversal is doubly incomplete: the transcript context menu offers "Name this speaker..." only on unnamed labels, with no "undo" / "rename back" / "remove from roster" on a named one; and the one existing lever, CLI `mp roster forget <name>`, removes the person from the roster but does not restore the transcript's original diarization labels (the relabel is baked into the file). Two consequences the owner hit: a merged cluster (two people sharing one `Unknown` cluster, the diarization-merge case) stays wrong with no recourse, and a wrong enrollment silently mislabels that voice in every future meeting through auto-naming, poisoning the roster with no discoverable fix. It also breaks the app's own reversibility model: text corrections are a reversible overlay (`TranscriptCorrectionStore`, where reverting drops the override), while speaker naming is a destructive mutation. Scope: (a) make the relabel a reversible overlay rather than an in-place rewrite, a speaker-label correction store paralleling `TranscriptCorrectionStore`, so the original diarization label is always recoverable and the display name resolves through the overlay at load; (b) an in-app affordance on a named speaker ("Undo naming" / "Rename..." / "Remove from roster") that reverts the transcript label and un-enrolls, or offers to un-enroll, the voiceprint through the existing `roster forget` path; (c) name-time honesty: the naming sheet states plainly that it enrolls a persistent voiceprint affecting future meetings and, until (b) lands, is not one-click reversible. Reuse the FEAT3-ROSTER relabel plumbing and `roster forget`; pure local roster + transcript ops, no new engine egress. This is the reversibility foundation FEAT3-SEGMENT rides on (per-segment reassignment through the same relabel path inherits the same one-way trap), so sequence it first. Priority note: filed P2 as the roster-correction family's most urgent member (a half-closed loop, naming with no un-naming), but it carries a P1-flavored angle, a wrong enrollment silently degrades every future meeting's speaker labels with no in-app recourse; promote to P1 if that silent-poisoning weighs heavier than the detection/storage P1 band. Acceptance: a named speaker reverts in-app to its original diarization label and can be removed from the roster; the original label survives the reversal (recoverable, not lost); a reverted enrollment no longer auto-names the voice in a later meeting; text-correction reversibility is unchanged.

**LANG1 (P2): summary language fidelity, verify the generated sections and cover the cloud path.** Owner-reported 2026-07-07 and confirmed by inspecting meeting `20260707-123334` (backend `anthropic`, model `claude-sonnet-4-6`): the transcript is entirely Latin/English (0 Cyrillic across 279 segments), yet the summary's `questions` and all ten `actions[].task` came out in Russian while title, summary, and decisions stayed English. The user uses English and Ukrainian and never Russian, so this is model-generated, not sourced from speech. Two root causes. (1) `summary_language = "auto"` tells the model to "write in the SAME language as the transcript, match the dominant language for code-switched content" (`_summary_language_directive` in `summarize.py`), which lets the model emit a language the transcript does not contain; here Claude localized the two audience-facing sections (open questions, action items) into Russian, apparently inferring a Slavic-speaking team from names and content. (2) The only drift safety net is local-only and blind to exactly these sections: `_retry_on_language_mismatch` lives in `summarize_local.py`, so the Anthropic path has no post-hoc language verification at all (a pre-write directive only), and even the local detector `_summary_detection_text` inspects title + summary + decisions + questions but omits `actions`, and in `auto` over a Latin-dominant transcript `_expected_summary_language` returns None so the check does not run. Net: neither backend verifies the action-items language, and the cloud backend verifies nothing. Scope: (a) extend the post-hoc language verification to every language-bearing generated field, including `actions[].task`; (b) make the verification backend-agnostic so a cloud summary whose sections diverge from the expected language is caught and repaired, not just the local one; (c) harden `auto`: when the transcript is confidently single-script (all-Latin, 0 Cyrillic), treat that as the target and flag a diverging section, since a Russian action block in an English meeting is an unambiguous divergence even under "auto"; (d) document a fixed `summary_language` as the safer setting. Interim mitigation (no code, owner-doable now): set `summary_language = "en"` (or `"uk"`) in `~/.config/meeting-pipe/config.toml`; the forced directive ("write in language `en` regardless of the transcript's language") removes the auto ambiguity. Sibling to LOCAL7 (whose local-only detector this generalizes) and LOCAL8; pure local verification over text the backend already produced, no new egress. Acceptance: a summary whose action items or questions diverge from the expected language is detected and repaired on both the anthropic and local backends; an all-English transcript never yields a Russian or other unexpected-language section; a fixed `summary_language` is honored across all sections; LOCAL7's existing coverage is unchanged for the sections it already handled.
