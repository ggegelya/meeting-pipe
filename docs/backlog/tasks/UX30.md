# UX30: Workflow delete flow (destructive affordance, dismissal, and the silent undo)

Band origin: filed 2026-07-23 (owner report of two defects in the workflow delete flow, plus three adjacent ones found while tracing it). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**UX30 (P2, new): deleting a workflow does not read as destructive, does not close its sheet, and can be silently undone by the next keystroke.** The owner reported the first two on 2026-07-23; the third is the one that makes this P2 rather than P3, because a confirmed destructive action can be reversed without the user knowing.

Reported defect 1, the Delete button does not read as destructive. `actionsRow` renders it as a neutral ghost capsule, `.buttonStyle(.mpGhost)` with a trash glyph (WorkflowsView.swift:570-575), so it carries the same visual weight as Save beside it. The comment at WorkflowsView.swift:567-569 justifies this as DSN28's move off colored inline buttons and asserts "there is no `.mpDanger` capsule". That assertion is stale: `MPDangerButtonStyle` exists and ships (BatchActionsPane.swift:504), used by the Library's own Danger card for "Move to Trash" (BatchActionsPane.swift:256), and the `.mpDanger` colour token is live in this very file (WorkflowsView.swift:104). So the app already has a destructive-button precedent for exactly this class of action, and the workflow editor is the surface that does not use it. Decide and record: promote `MPDangerButtonStyle` out of its private scope into `Preferences/PreferencesControls.swift` (the documented home for shared primitives per daemon/CLAUDE.md) and use it here, matching BatchActionsPane, or keep the ghost capsule and carry the emphasis on the label and glyph via `.mpDanger`. Either resolution must also correct the stale comment, since the next reader will otherwise re-derive the same wrong premise.

Reported defect 2, the sheet stays open after a confirmed delete. `deleteSelf` calls `store.delete` and nothing else (WorkflowsView.swift:644-650), while `save()` ends with `onCommit?()` (WorkflowsView.swift:638). `onCommit` is documented at WorkflowsView.swift:49-51 as the hook "called after a successful Save so the host sheet can dismiss", and both hosts wire it to their close path: the edit sheet at LibraryWindow.swift:655 (`onCommit: onClose`) and the new-workflow sheet at LibraryWindow.swift:699 (`onCommit: { onClose(s.id) }`). Delete never reaches it, so after confirming, the user is left editing a workflow that no longer exists on disk, in both sheets.

Found defect 3, the confirmed delete is silently undone by Save. This follows directly from defect 2 and is the reason to treat the pair as correctness. The stranded sheet still holds the full form state, and Save carries `.keyboardShortcut(.defaultAction)` (WorkflowsView.swift:565), so one Return (the reflex after any dialog) calls `save()`, which calls `store.upsert(clone)`; `upsert` finds no matching id, takes its append branch, and persists (WorkflowStore.swift:63-68). The workflow the user just confirmed deleting is written back to disk, with no message either way. In the new-workflow sheet the same path resurrects a stub the editor's Delete had just removed.

Found defect 4, a refused delete is indistinguishable from a successful one. `deleteSelf` discards the return value (`_ = try store.delete(id: workflow.id)`), and `delete` returns `false` without throwing when the id is not found or the workflow is the default (WorkflowStore.swift:73-75). Both cases close the alert and leave the UI unchanged, so a no-op looks exactly like a delete. The default-workflow case is currently unreachable from the button (`.disabled(workflow.isDefault)`, WorkflowsView.swift:576), but that disabled control gives no reason for being disabled, which is its own small truthful-states gap.

Found defect 5, the confirmation copy understates the consequence. The alert says "Meetings already recorded keep their workflow metadata; only future meetings stop matching this workflow" (WorkflowsView.swift:119). The sidecar half is true, the rail half is not: `LibraryScope.resolveWorkflow` resolves a meeting to a workflow by NAME (LibraryScope.swift:112-116, the deliberate TECH-B4 bridge because the sidecar persists the name for cross-machine portability) while scopes are tagged by id (LibraryScope.swift:23, :107-108). After a delete, every past meeting whose sidecar names it resolves to nil, so it matches no workflow scope, and it is not swept into Untagged either, because `.untagged` tests for an absent or empty name (LibraryScope.swift:101-102) and the name is still present. Those meetings therefore vanish from the WORKFLOWS rail entirely while appearing nowhere that explains why. Either the copy says so, or the fall-through is fixed so a meeting naming a workflow that no longer exists reads as untagged; prefer the copy change, since altering filter semantics reaches further than this task.

WF9 interaction, cross-referenced rather than duplicated. The same name-based resolution means the owner's four same-named "Client work" workflows all resolve to whichever is first by name, so three of those four rail scopes necessarily count zero meetings regardless of which one recorded them, and deleting one can silently change which workflow every "Client work" meeting resolves to. That is context for WF9's cleanup, not a second cause-hunt here. One lead worth recording for WF9's pickup: the new-workflow sheet persists its stub on open and relies on Cancel to remove it (LibraryWindow.swift:686-692), so any dismissal that does not run Cancel orphans an "Untitled workflow", and the owner's config contains exactly one such row; consistent with that path, not proof of it.

Scope, smallest change per defect: `deleteSelf` invokes `onCommit?()` on a true return; a false or throwing delete surfaces the reason in the existing `saveError` slot instead of closing silently; a confirmed delete cannot be reversed by a later Save in the same sheet (dismissal is the primary fix, plus a belt at the editor seam, either a deleted latch that disables Save or an upsert that refuses an unknown id from an editor context; pick one and pin it); the destructive-affordance decision above, including the stale comment; a reason on the disabled-for-default Delete; the alert copy correction.

Explicitly not: WF9's duplication cause-hunt and the owner's manual cleanup; no undo or trash for workflows (a confirmation plus honest copy is the register, and the meta sidecar already preserves recorded meetings); no change to `WorkflowStore.delete`'s refusal to remove the default workflow (a correct invariant, only its silence is in scope); no re-opening DSN28's direction beyond this one destructive control; no change to name-based meeting resolution (TECH-B4's portability call).

Acceptance: Swift tests pin that a successful editor delete invokes the host's commit hook exactly once and a refused delete does not, and that a refused or failed delete leaves a visible reason; a test pins that a delete followed by a save cannot re-persist the deleted workflow (whichever belt is chosen); a presentation test pins the destructive style on the Delete control and the reason on the disabled-for-default state. If `MPDangerButtonStyle` is promoted, it lands in `Preferences/PreferencesControls.swift` and BatchActionsPane consumes the shared one, so no parallel copy exists. The stale comment at WorkflowsView.swift:567-569 and the alert copy both reflect what the code does after the change. Full daemon suite green, `swift build` clean. Owner eyeball: the Delete control reads as destructive, confirming closes the sheet, the workflow is gone from the rail, and pressing Return afterwards does not bring it back.
