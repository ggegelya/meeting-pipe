# PIPE10: Sleep-resilient summarize retry

Band origin: filed 2026-07-17 (from the 2026-07-16 sleep incident). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**PIPE10 (P2, new): sleep-resilient summarize (retry once after a slept-through timeout).** Filed from the 2026-07-16 incident (stem `20260716-163031`, a 110-min Teams meeting). The lid closed one minute after the meeting ended, so post-meeting processing froze under clamshell sleep; transcription finished across dark-wake slivers, and the Anthropic call left during a 21 s dark wake at 19:57:48. The Mac then slept through the request: the 120 s httpx read timeout (`summarize._REQUEST_TIMEOUT`) rides the monotonic clock, which freezes during sleep, so the budget burned away across dark-wake slivers (the run's 5 s progress beats reached `elapsed_s: 123` over 2 h 13 m of wall time) and fired under a minute after the 22:10 lid-open wake as `APITimeoutError`. That error is deliberately excluded from tenacity retries (`_should_retry_anthropic`: "A stall should drop straight to the backend fallback (local) instead"), but the fallback (`_AutoFallbackClient`) exists only for backend `auto`; this meeting's workflow pins `anthropic`, so the exclusion's own rationale does not hold there and the run hard-failed with `<stem>.error.json` stage=pipeline. The manual Library Retry then succeeded in 36 s, proving the request itself was fine. The class: any pinned-cloud meeting whose summarize is in flight when the Mac sleeps is guaranteed to fail and then waits, sidecar-failed, for a manual click.

Scope: detect slept-through-it and retry once, in-process, at the `_create_message` seam. Capture both clocks at attempt start (`time.time` wall, `time.monotonic`); on `APITimeoutError`, when wall-clock elapsed exceeds monotonic elapsed by a clear margin (>= 60 s, i.e. the Mac demonstrably slept mid-request), log the divergence and re-issue the request once, fresh; a second timeout, or a genuine stall with agreeing clocks, fails exactly as today. Prefer a small wrapper loop around the SDK call inside `_create_message` so the tenacity predicate and the transient-retry classes stay untouched. The same hole exists in principle for pinned-cloud `engine.complete_text` paths and the `openai` provider; fix summarize first and note in the ship whether the helper is worth promoting (`backend_fallback` already absorbs the class for `auto`).

Explicitly not the fix: extending REC8's idle-sleep assertion to the processing queue would not have helped (clamshell sleep on battery overrides assertions), and flipping the workflow to `auto` changes summary-quality behavior (silent local fallback) rather than fixing the class. Observed in the same incident, out of scope here (file separately if it recurs): the Library row keeps showing Failed for the ~40 s between the Retry click and `run_started`, which reads as "retry did nothing".

Acceptance: a unit test with monkeypatched clocks pins the three paths (slept-through timeout retries exactly once and can succeed; genuine-stall timeout fails immediately as today; non-timeout errors keep their current retry classes); the retry logs a line naming the wall-vs-monotonic divergence so the next investigation needs no pmset archaeology; pipeline suite + ruff + pyright green.
