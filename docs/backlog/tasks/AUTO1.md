# AUTO1: Automation surface: App Intents + URL scheme

Band origin: feature review 2026-07 (owner approved 2026-07-05). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**AUTO1 (P2, promoted from P3 on 2026-07-10): automation surface, App Intents + URL scheme.** Promoted by the architecture review as the highest-leverage de-CLI enabler (Shortcuts, Raycast, and Stream Deck ride for free). The daemon is driveable only by two hotkeys and the menu bar. Expose start/stop/toggle record (with a BYO variant), open Library (optionally scoped), ask, and digest via App Intents (Shortcuts) and a `meetingpipe://` URL scheme, so Raycast and Stream Deck ride for free. External triggers respect exactly the gates the hotkey does (mic permission, cooldowns, no start while recording); local IPC only, no new egress. Acceptance: a Shortcut toggles a recording end-to-end and a denied-mic trigger routes to the Permissions tab instead of silently failing.

**Progress (2026-07-11): URL scheme shipped and verified; App Intents deferred on the toolchain.** The `meetingpipe://` half is done. `Automation/AutomationCommand.swift` is a pure, unit-tested parser (`toggle` / `record[?byo]` / `stop` / `library[?scope=ask|digests|facts]` / `ask?q=` / `digest`); `AppDelegate.application(_:open:)` receives the deeplink (Launch Services delivers it to the running daemon once `CFBundleURLTypes` is registered), and `Coordinator+Automation.handleAutomation` routes it through the same `session` entry points the hotkey uses, so a denied mic still deeplinks to Permissions (via `beginRecording`) and `record` only ever starts from idle (never stacks a second recording). `library` / `ask` / `digest` land at the right Library rail via new `pendingScope` / `pendingAskQuestion` on `LibraryWindowModel` (`ask` prefills + runs the question, `digest` triggers `generateDigest`). `CFBundleURLTypes` (scheme `meetingpipe`) is in `scripts/install.sh`'s Info.plist heredoc. This meets the acceptance: a Shortcut using the built-in "Open URL" action with `meetingpipe://toggle` toggles a recording end-to-end, and Raycast / Stream Deck ride the same way with no App Intents metadata. Verified: daemon 1121 green (6 new `AutomationCommandTests`). Two remainders keep this partial: (a) **native App Intents are not shipped** (a first-class, no-URL-typing Shortcuts action), because the daemon builds with a plain `swift build` and no Xcode project, which does not run the `appintentsmetadataprocessor` build phase Shortcuts needs to discover App Intents, so shipping them now would be undiscoverable dead code; lighting them up means either moving the build to Xcode or invoking the metadata processor in `install.sh`, a separate spike that DIST1's build-system question may force anyway. (b) **Owner-owed runtime eyeball**: confirm `open meetingpipe://toggle` actually routes on a real Mac (Launch Services URL delivery is not exercisable in this harness, and the menu-bar app is not screenshotable here), and note the one-time `scripts/install.sh` re-run to register the scheme on an existing install (the fast `rebuild.sh` does not rewrite Info.plist).

**Progress (2026-07-18): App Intents leg spiked probe-first; the metadata-processor blocker is lifted.** Remainder (a) turned out to be answerable in-harness, so it shipped as a probe-first spike (the MIC7 / DET2 / MIC8 pattern): [`docs/spikes/auto1-app-intents-metadata.md`](../spikes/auto1-app-intents-metadata.md) (analysis + verdict) and [`daemon/scripts/auto1-app-intents-probe.sh`](../../daemon/scripts/auto1-app-intents-probe.sh) (a self-contained, owner-run mechanism check). The spec assumed the metadata phase was out of reach off SwiftPM; it is not. Full Xcode is present, and the probe drives the exact escape hatch install.sh would use, a supplementary `swift-frontend` const-values pass over the App Intents sources (`-emit-const-values-path` + a bare-name `-const-gather-protocols-file`, absolute source paths, `-parse-as-library`) fed to `appintentsmetadataprocessor`, and it verified end-to-end that this emits a valid `Metadata.appintents` bundle naming the intent, its title, and its App Shortcut phrase on Xcode 17F42 / macOS 26 SDK (actionsdata format 3.0). So the App Intents build is now **bounded, not blind**: N `AppIntent` structs mirroring the six verbs + one `AppShortcutsProvider` + one install.sh metadata step, each `perform()` posting the matching `meetingpipe://` URL so the shipped, tested `handleAutomation` gate stays the single path. Verdict: **mechanism GO (proven here), build timing the owner's call, discovery eyeball owner-owed.** Nothing was added to the shipping build, because the URL scheme already meets the acceptance and the metadata step is a private-flag surface that can drift on a toolchain bump (the probe is the regression guard) and whose durable home is whatever build system DIST1 blesses, so it should converge with DIST1 rather than be bolted on ahead of it. The task stays **partial** with (a) sharpened from "blocked, unknown mechanism" to "bounded build on owner go-ahead + the discovery eyeball", and (b) unchanged (owner-owed runtime).

**Progress (2026-07-19): native App Intents built; the task drops to owner-owed.** The owner ran the probe on their Mac (`MECHANISM GO`) and gave the go-ahead, so remainder (a) is now shipped rather than queued behind DIST1. Three pieces: `Automation/MeetingPipeAppIntents.swift` (six intents, `ToggleRecording` / `StartRecording` with a BYO parameter / `StopRecording` / `OpenLibrary` with a rail `AppEnum` / `AskMeetingPipe` with a question / `GenerateDigest`, plus a `MeetingPipeShortcuts: AppShortcutsProvider` giving toggle / stop / digest ready-made voice phrases); `MeetingPipeAppIntentsTests` (8 tests); and the extraction step in `scripts/install.sh`. Two design decisions worth recording. **First, every `perform()` executes by opening its own `meetingpipe://` URL** rather than calling the coordinator directly, so all six actions inherit the shipped, unit-tested `handleAutomation` gates (mic-denied deeplinks to Permissions, `record` never stacks) and there is exactly one gate path, not two. **Second, the intents file is deliberately dependency-free**, referencing no other type in the module, because `install.sh` compiles it STANDALONE with `swift-frontend` to emit the const-values and a lone-file compile cannot resolve cross-file references; the cost is a second copy of the verb vocabulary, and the mitigation is `MeetingPipeAppIntentsTests` parsing every intent's URL back through `AutomationCommand.parse`, so a renamed verb or dropped query key fails a test instead of shipping an action that silently does nothing. That trap is written into `daemon/CLAUDE.md`'s Don't list, because the failure mode is silent in the worst way (build succeeds, app installs, only the Shortcuts actions vanish). The install.sh step is non-fatal by design (a missing full Xcode warns and skips, since the URL scheme still works) and runs before the signing pass so the metadata is covered by the signature; `Contents/Resources/Metadata.appintents` is the correct location, confirmed against ten shipped Apple and Microsoft apps rather than guessed. Verified here: `swift build` clean under CONC4's strict concurrency, 14 automation tests green, the real production file extracting all six intents + three phrases, and the install.sh block executed against a staged bundle. What is left is **only remainder (b), the owner runtime eyeball**, now covering both layers: run `scripts/install.sh` (a fast `rebuild.sh` writes neither the Info.plist nor the metadata), confirm the six actions appear in Shortcuts.app and that one toggles a recording end-to-end. Status therefore moves `partial` to `owner-owed`. **(Corrected the next day, see below: the App Intents leg is discoverable but not invocable, so this note's "shipped" framing was premature.)** The DIST1 convergence note stands: if DIST1 adopts an Xcode project, the metadata phase comes for free and the install.sh hand-roll retires; until then `daemon/scripts/auto1-app-intents-probe.sh` is the guard to re-run after an Xcode bump, since the extraction rides private compiler flags.

**Progress (2026-07-20): the App Intents leg is discoverable but NOT invocable; blocked on code signing, and the URL half is now hardware-verified.** The owner installed the build and reported the actions failing. Measured outcome: all six actions **list correctly in Shortcuts** with their parameters (so the metadata pipeline this task built is genuinely correct end to end, verified again against the installed bundle), but **invoking any of them fails** with "Shortcuts couldn't communicate with the app". The app is never contacted: no crash report, the daemon process is untouched, and no `automation` event is logged, so `perform()` never runs.

Six candidate causes were each excluded by direct measurement, recorded here so nobody re-runs them: (1) the intent types ARE in the shipped binary (AppIntents linked with 95 symbol refs, `__swift5_proto` conformance sections present, not dead-stripped); (2) Launch Services DOES register the app (`lsappinfo` lists it with its bundle id); (3) the failure is identical with `openAppWhenRun` set to `true` and to `false`; (4) identical whether launchd execs the binary or Launch Services opens it (`open -a` after a `bootout`); (5) identical with `LSUIElement` `true` and `false` (tested by rewriting the installed Info.plist and re-signing); (6) the app never crashes. One earlier suspect was also disproved by a control rather than trusted: Launch Services reports `Intents: 0` for our bundle versus `1` for Weather.app, which looked damning until Microsoft Word (working App Intents) also reported `0`, so that field tracks SiriKit `NSUserActivityTypes`, not App Intents.

What is left unexcluded is **code signing**: the bundle is `spctl`-rejected with `TeamIdentifier=not set`, uniquely among third-party apps on the dogfood Mac (Ghostty, Obsidian, Notion all pass with real team ids). Discovery reads a static file and needs no trust; the intent handoff is an XPC path that does. That is consistent with every observation, including why `meetingpipe://` keeps working. It is a strong inference rather than a proof, because confirming it needs a Developer ID this repo does not have, and the system log is unreadable in the harness (`log show` returns zero rows), so the refusal could not be read directly.

Decision: **`install.sh` emits the metadata only under `MP_APP_INTENTS=1`, default off.** Shipping it would put six actions in the owner's Shortcuts library that every click fails on, which is a worse product than not offering them; the de-CLI rule wants a working affordance, not a visible-but-broken one. The code, the 8 drift-fence tests, and the extraction step all stay, so the leg is one signed build away. The remainder is therefore **gated on D8 / DIST1 (a real Developer ID)** rather than on further engineering in AUTO1, and D8 gains a concrete user-visible payoff it did not have when it was parked ("deferred until a second user"): native Shortcuts / Spotlight / Siri actions.

Two things did close, both worth keeping. **The URL scheme is now verified on real hardware**: firing `open meetingpipe://library` produced `automation command` with verb `open_library` in `events.jsonl`, which closes remainder (b), the owner-owed routing eyeball that had been open since 2026-07-11. AUTO1's stated acceptance is met by that layer. **And a latent install bug was found and fixed**: `install.sh` never actually restarted the daemon, because a failed `launchctl unload` was swallowed by `|| true` and the later `load -w` is a no-op when the job is already loaded, so every install since the LaunchAgent landed left the PREVIOUS binary running while reporting success (caught here as a daemon from 22:50 surviving a 23:14 install). It now `kickstart -k`s the job, verified by the `app_rebuild` cdhash change in the event log. That defect was silently undermining every "I installed it and nothing changed" observation, not just this task.

Status returns to `partial` (not `owner-owed`): the owner has nothing left to look at, the block is a missing Developer ID.
