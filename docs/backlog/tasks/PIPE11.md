# PIPE11: Regenerate correctness: same-stem serialization, provenance, policy snapshot

Band origin: filed 2026-07-23 (event-log forensics on stem `20260723-103118`, 2026-07-23 assessment session). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**PIPE11 (P2, new): a regenerate can race a concurrent writer, lies about its engine afterwards, and can straddle a policy change mid-chain.** Three defects, one path, all observed in production logs on 2026-07-23.

Context, the observed incident. The regenerate path (`MeetingLibraryService.regenerateMeeting`, Library/MeetingLibraryService.swift:124-159) spawns `mp summarize [--backend X]` directly (`PipelineLauncher.summarize`, PipelineLauncher.swift:477-485) and never enters the `SinkDispatcher` `ProcessingJob` queue, so it runs concurrently with an active run-all and with other regenerates for the same stem; nothing keys on stem to block, drop, or coalesce. On 2026-07-23 (all times UTC): 07:46:15 Reprocess enqueued a run-all; 07:46:26 a regenerate spawned concurrently against the same stem; 07:48:43 a second regenerate spawned (NDA still on, clamped local); 07:49:00 the workflow was reassigned to General (nda=false); 07:49:06 a third regenerate spawned (now genuinely Anthropic). The second (local) writer finished ~9 s before the third (Anthropic) one; had the ordering flipped, a stale local summary would have silently overwritten the Sonnet summary the owner had already seen. Notion received the stale local content first (07:49:19, page slug `Daily-Stand-up-Meeting-Summary`) and the Anthropic upsert 14 s later (07:49:35, slug `Daily-Stand-Up-DMS-5-12-Hotfix-Auth-Epic`), same page id. Writes are individually atomic (PIPE8's `storage.atomic_write_text`), so this is last-writer-wins, not corruption; the defect is that "last" is a coin flip.

Defect 2, provenance. `<stem>.run.json` (backend, model) is written only by `orchestrate.run_all` via `corrections.write_run_sidecar` (pipeline/src/mp/corrections.py:51); a standalone `mp summarize` returns `SummaryOutput` with the true `backend`/`model` (summarize.py:469-476) and throws it away. After the 07:49 Anthropic regenerate, `run.json` on disk still read `backend=local, model=mlx-community/Qwen2.5-7B-Instruct-4bit, ts=07:48:04`, so the Library detail header showed "On-device (MLX)" over a Sonnet summary (`MeetingStore` reads run.json backend at MeetingStore.swift:635). The same stale attribution poisons the corrections corpus LOCAL9 trains on (records resolve backend from the run sidecar).

Defect 3, policy snapshot. Each spawned subprocess re-reads `<stem>.meta.json` at its own `entry.prepare`, so a regenerate-then-republish chain started under one policy can publish under another: the 07:48:43 regenerate summarized under the NDA clamp (local backend, filesystem sinks), but its republish leg spawned at 07:49:18, after the reassign, re-read the now-General meta and pushed that stale local summary to cloud Notion. Direction of the leak was benign here (the owner had just de-NDA'd deliberately); the inverse direction is the one to close cheaply: a chain started pre-NDA must not egress a summary after the meeting was moved INTO an NDA workflow mid-chain.

Scope. (a) Per-stem serialization: route Library regenerate/republish through the `SinkDispatcher` queue as a lightweight job kind (preferred: it buys UX28's `activeProcessing` visibility and run-all's cancel for free and keeps one execution lane per stem), or, if the queue coupling proves heavy, a per-stem async mutex in `MeetingLibraryService` with coalescing (a second identical request while one is pending is dropped with an event). Same-stem ordering is the invariant; cross-stem concurrency stays as-is. (b) Provenance: standalone `mp summarize` rewrites the run sidecar on success with the `SummaryOutput` backend/model (reuse `corrections.write_run_sidecar`; preserve the existing `publish_state` via `corrections.set_publish_state` semantics rather than resetting it), and a `--candidate` run (TECH-A16 preview) explicitly does not touch it; the Swift badge then stays truthful with no daemon change. (c) Policy: with (a) in place the reassign action joins the same per-stem lane (reassign is already a Library action), which closes the TOCTOU structurally; the acceptance scenario below pins the INTO-NDA direction regardless of which mechanism lands.

Explicitly not: no changes to `config.effective_backend` or the PIPE6 one-shot override semantics (the NDA clamp is deliberate and stays); no cross-stem global lock; no UI work (UX28 owns surfaces; this task is correctness even if UX28 never lands); no rewrite of `publish_router` idempotency (it already upserts correctly once ordering is fixed).

Acceptance. Pipeline: a test pins that `mp summarize` on success rewrites run.json with the answering backend/model and leaves `publish_state` intact, and that `--candidate` leaves run.json untouched; ruff + pyright + pytest green. Daemon: a test pins that a regenerate requested while a same-stem job is active does not spawn concurrently (queued or coalesced, with the event emitted), and that requests for different stems still overlap; a scenario test pins that a chain started before a reassign INTO an NDA workflow cannot publish to a cloud sink after the reassign (fake launcher + meta rewrite mid-chain). Events: reuse the existing coordinator category for the queued/coalesced action; CONVENTIONS table row in the same commit if a new action name is added. Full daemon suite green. Owner eyeball: after an Anthropic re-summarize of an NDA-turned-General meeting, the detail header badge reads Anthropic, and run.json matches.
