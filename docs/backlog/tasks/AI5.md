# AI5: Spike: meeting-type classification

Band origin: intelligence layer (AI band). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**AI5 (P3, spike): meeting-type classification.** Prototype on-device meeting-type labels over the existing library (a short local call, or a heuristic over attendees / cadence / title) and show the results to the owner before committing a permanent `MeetingSummary` field. Uncommon among competitors (only MeetGeek auto-detects). Acceptance: a throwaway classifier labels the library by type; the spike returns a recommendation on whether the labels are useful enough to wire into grouping/digests and earn a schema field; no schema change lands until that decision.

**Progress (2026-07-11): prototype shipped, usefulness judgment owner-owed.** `mp classify-meetings` (`classify.py`) discovers every `<stem>.summary.json` and labels it against a fixed 10-type taxonomy. Default is a zero-egress heuristic over the title, workflow name, and attendee count (first-match keyword rules, then attendee-count fallbacks); `--llm` adds a per-meeting label from `engine.complete_text` beside it, routed through `effective_backend` so a regulated/NDA config stays on-device (the AI/DV standing rule), plus a `--json` payload. It arms the egress guard via `entry.prepare` (secrets only under `--llm`). CLI-only is intentional (a spike harness, de-CLI exempt like dogfood / analyze-detection); registered in `__main__.py` + USAGE + the ARCHITECTURE subcommand table. The pure heuristic, discovery, LLM-snapping, and report are pinned by 8 `test_classify.py` tests (698 pipeline green, ruff + pyright clean); a live heuristic run on this Mac confirmed the end-to-end flow, but the library is only 2 test meetings (both landed `client`), too thin to judge. Owner-owed remainder, the spike's headline question: run `mp classify-meetings [--llm]` on the real, diverse library and decide whether the labels cluster usefully enough to wire into grouping/digests and earn a permanent `MeetingSummary.meeting_type` field. Recommendation if adopted: heuristic-first (cheap, zero-egress, no model) with the LLM as a fallback/refinement, not an always-LLM field. No schema change landed (acceptance honored).
