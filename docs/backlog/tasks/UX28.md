# UX28: Library processing visibility and control

Band origin: filed 2026-07-23 (owner findings, 2026-07-23 assessment session; event-log forensics on stem `20260723-103118`). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**UX28 (P2, new): every Library-triggered job is visible while it runs and stoppable when it should not be.** The owner hit all three gaps in one session on 2026-07-23: a mistaken Reprocess could not be cancelled for its whole ~110 s run, a Re-summarize clicked mid-reprocess produced no visible effect at all ("It looks like nothing happened"), and nothing distinguished a queued job from a finished one. PRODUCT.md's truthful-states principle is the bar; PIPE10's out-of-scope note already anticipated this class ("the Library row keeps showing Failed for the ~40 s between the Retry click and run_started, which reads as retry did nothing").

Context, the verified gaps (line refs as of 2026-07-23). (1) Cancel plumbing exists end to end (`LibraryWindowModel.cancelProcessing` LibraryWindow.swift:237-239 -> `Coordinator+MenuActions.swift:94-96` -> `SinkDispatcher.cancelActiveJob` SinkDispatcher.swift:212-216 -> `PipelineLauncher.cancelActiveRun` PipelineLauncher.swift:276-284, SIGTERM then SIGKILL after 10 s, job completes as a retryable failure) but the only UI that reaches it renders under a stalled heartbeat: `MeetingRow.processingIndicator` shows the Cancel button only when `p.stalled` (MeetingRow.swift:337-346, 30 s threshold at SinkDispatcher.swift:37). A healthy job shows an elapsed-time pill and nothing else. (2) Queued-but-not-started jobs have no visible state and cannot be removed: `SinkDispatcher.enqueue` appends unconditionally (SinkDispatcher.swift:61-72) and the only dequeue is `removeFirst()` inside `completeActiveJob` (SinkDispatcher.swift:179-181). A second Reprocess click silently enqueues a duplicate. (3) The regenerate/republish path publishes no state anywhere: `activeProcessing` (LibraryWindow.swift:72-77) is fed only by run-all progress callbacks (Coordinator.swift:300-319); the row spinner `inFlight` is view-local `@State` set only by the row context menu's own actions (MeetingRow.swift:31, 468-491); the detail-menu "Re-summarize with ..." (`MeetingDetailView+Header.swift:407-411`) fires a fire-and-forget Task with no spinner, no toast, no completion signal, and only one variant alerts on failure (SummaryTab.swift:487-492). On 2026-07-23 the regenerate's only observable effect was an `idempotent: true` republish 100 s after the click.

Scope, five legs, smallest-change each. (a) Always-visible Cancel for the active job: the row's processing indicator and the detail header surface Cancel whenever `activeProcessing` matches the stem, healthy or stalled, wired to the existing `cancelProcessing()`; keep the Stalled pill semantics unchanged. (b) Pending-queue truth: queued stems render a quiet "Queued" state (the row already receives `activeProcessing`; extend the published surface with the pending stems from `SinkDispatcher.processingJobs`) and a dequeue affordance removes a not-yet-started job (new `SinkDispatcher.removePending(id:)`, emitting a `pipeline_dequeued` event on the existing coordinator category). (c) Published per-stem busy state for regenerate/republish: replace the view-local `inFlight` with a `@Published` per-stem state set by `MeetingLibraryService.regenerateMeeting`/`republishMeeting` so BOTH affordances (detail menu and row context menu, plus SummaryTab's variant) render the same "Summarizing with Anthropic..." / "Publishing..." chip in the row and the detail header; label carries the requested engine. (d) Completion and failure feedback: regenerate/republish completion routes through the same surface the run-all queue already uses (`PipelineJobDispatcher`'s done-notification / error-banner closures) so success is visible from anywhere and failure is not silent; unify the one-variant NSAlert. (e) Cancellable regenerate: `PipelineLauncher` retains spawned summarize/publish processes per stem (today `retainForCancel: true` is passed only by `runAll`, PipelineLauncher.swift:694, 312) so the busy chip from (c) can offer Cancel; termination follows the existing SIGTERM-then-SIGKILL shape and surfaces as a normal failure with retry.

Interplay: PIPE11 (filed the same day) owns same-stem serialization and provenance; if PIPE11 routes regenerate through the SinkDispatcher queue, leg (c) collapses into the `activeProcessing` surface and leg (e) inherits run-all's existing cancel; whichever task lands second reconciles, and the acceptance bars below hold either way.

Explicitly not: no progress percentages (stage + elapsed from the existing heartbeat is the register); no dedupe/coalescing logic (PIPE11); no NDA-specific affordances (UX29); no new config knobs; no cancel for the in-process ANE transcription stage (it is not a subprocess; out of scope until it hurts, note it in the ship if the boundary is visible in testing).

Acceptance. Swift tests pin: the presentation model shows Cancel for a healthy active job and for a stalled one; a pending job renders Queued and `removePending` drops it exactly once with the event emitted; the per-stem busy state is set and cleared by regenerate and republish from both affordances (fake launcher), including the failure path; a cancelled regenerate completes as a retryable failure and clears the busy state. Events: `pipeline_cancelled` (existing) fires from the always-visible button; `pipeline_dequeued` is new on the existing category (CONVENTIONS event-table row in the same commit; `scripts/truth_fences.py events` green). Full daemon suite green; no pipeline-side changes. Owner eyeball: click Reprocess on a healthy meeting and cancel it within 5 s; click Re-summarize from the detail menu and watch the chip appear, complete, and toast.
