# LOCAL14: Staged local summarization pipeline (extract, compose, verify)

Band origin: filed 2026-07-23 (owner ask from the 2026-07-23 local-quality assessment; the owner's direction verbatim: raise local quality via architecture, multi-run passes, consensus, caching; cloud parity explicitly not the target; open to architectural change). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**LOCAL14 (P2, new): replace the local backend's single do-everything call with a staged pipeline: grounded extraction, per-section composition, deterministic-first verification.** Do LOCAL13 (model refresh) first; a staged pipeline on a better 8-12B compounds, and clean attribution requires changing one lever at a time.

Context, why the single call caps quality. `LocalSummaryClient.summarize` asks one small model, in one shot, to read the whole transcript, obey a ~60-line defensive master prompt plus a full JSON schema pasted as text, and emit title + summary + decisions + actions + questions + attendees at once (summarize_local.py:250-308, `_augment_with_schema` :633-667). Three structural consequences, all visible in the 2026-07-23 side-by-side on stem `20260723-103118`: (1) the defensive rules (no inventions, default-to-empty) are correct hallucination control that a 7B over-obeys into genericity ("Discussed deployment of hotfix and assigned tasks"); (2) JSON discipline competes with content for the model's attention, which is why the 3-layer extractor and the schema-violation replay exist at all (:379-407, :739-776); (3) for long meetings the PIPE4 map-reduce summarizes 1,500-char windows into partial summaries and then summarizes the summaries (:88-98, :310-377), a double abstraction that compounds vagueness. Small models are markedly better at grounded extraction (find and quote) than at open abstraction (synthesize everything); the design below moves the abstraction burden onto short, dense, already-extracted context.

The staged design, refining the owner's sketch (numbered passes: general summary, key areas, decisions, then re-iterations, consensus voting, prompt caching, automatic learning). Stage 1 EXTRACT: map over transcript windows sized to the model's real budget (model-aware, not the current worst-case 1,500 chars) and pull atomic facts as structured rows: commitments, requests, dates, tickets and artifacts, self-identifications and name cues, unresolved questions, each with a verbatim quote and [mm:ss] timestamp. Extraction is the easy shape for a small model, and the quotes are FEAT10's evidence anchors for free. Stage 2 COMPOSE: one small call per section (summary bullets, decisions, actions, questions, title), each fed ONLY the extracted fact rows, never the raw transcript; a section prompt can then demand specificity (who + what + artifact per bullet) without starving output, because the facts list is short and dense. Stage 3 VERIFY: deterministic checks first (every decision and action must trace to a quote that resolves in the transcript, the FEAT10 verifier shape; owner names must appear in a supporting quote; caps enforced; dedupe), and one LLM critic call only for what determinism cannot judge (e.g. is this a commitment or an intention), with the master prompt's worked examples as its rubric. On the owner's re-iteration passes (sketch steps 4-5): fold them into Stage 3's single critic pass; open-ended self-refine at 7-14B tends to paraphrase rather than improve, so iterate only if the dogfood bench proves a second pass pays. CONSENSUS, bounded: k=3 samples at temperature ~0.6 for the two contested fields only (the decisions list, action owners), majority-keep; adopt only if it beats the critic alone on the bench, and never for the full summary (expensive, and averaging prose regresses to mush). PROMPT CACHING: keep every stage's system+schema prefix byte-stable and put volatile content last so a prefix cache can bite; measure whether the installed `mlx_lm.server` reuses KV across requests at all, and if not either accept the cost (windows are small) or move the loop in-process (ARCH5 already names the LocalSummaryClient split seam; `docs/spikes/local12_bench.py` is the latency harness). A warm server is reused across stages (LOCAL11's identity check applies). Per-stage minimal JSON schemas replace the one giant schema, and if LOCAL13's probe shows `response_format` is honored, lean on it per stage.

Interplay. FEAT10: build Stage 1 quotes+timestamps and the Stage 3 resolver AS FEAT10's local evidence leg, one design landing both, and the FEAT10 spec's "local map-reduce degrades to no evidence" caveat dissolves because this pipeline replaces that path. LANG1/LOCAL7: the language check runs on the final composed summary exactly as today; quotes stay verbatim transcript language (FEAT10's divergence exclusion applies). Long meetings: extraction IS the map, composition reduces facts rather than partial summaries, so the PIPE4 reduce path in `summarize_local` is subsumed; the cloud paste-bundle guard (`skip_above_chars` on cloud backends) is untouched. Degenerate transcripts (LOCAL2 quality checks) short-circuit to the current single-pass shape, which stays in the code as the fallback while the bake-off runs. Automatic learning on corrections stays LOCAL9's (its `mp train-adapter` + dogfood adoption verdict); a retrain cadence is premature until that verdict lands.

Gate, spike discipline. Adopt staged-vs-single-pass on the SAME model via `mp dogfood`, thresholds pre-registered before scoring: actions_capture and decisions_capture must improve by a margin the owner sets in the doc, hallucination_rate must not regress (the verifier should improve it), wall-clock per 15-minute meeting under an owner-set ceiling, RAM inside the caffeinate + free-RAM-watchdog protocol. Keep single-pass as the fallback path; the staged path becomes default only on a verdict.

Explicitly not: cloud backends keep their single tool-use call (Anthropic does not need this and the paste guard economics differ); Apple Intelligence untouched (experimental backend, LOCAL8); no new user-facing config knobs (one internal toggle may exist during the bake-off and dies at adoption, the HYG2 rule); no automatic LoRA retraining (LOCAL9); no streaming/live summarization (I8).

Acceptance. Unit tests against a scripted fake server pin: extraction output shape (quote + timestamp per fact row); composition receives fact rows only (a probe asserting the raw transcript never appears in Stage 2 messages); the verifier drops a fabricated quote, an unresolvable timestamp, and an owner with no supporting quote; the consensus majority rule (if adopted); the degenerate-transcript short-circuit. The dogfood report with pre-registered thresholds and the verdict is committed; latency and peak RSS recorded per stage. ruff + pyright + pytest green; ARCHITECTURE.md's summarize_local description and CONVENTIONS updated in the same commit that flips the default. Owner eyeball: a regenerated `20260723-103118` local summary names people, tickets, and artifacts concretely enough to be checked against the transcript via its FEAT10 chips.
