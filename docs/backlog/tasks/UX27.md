# UX27: Floating-panel keyboard operability

Band origin: deferred out of UX23 (owner feedback 2026-07-22). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**UX27 (P4): make the detection prompt and recording HUD keyboard-operable.** PRODUCT.md's a11y floor asks for "the prompt panel's Record / Skip / BYO buttons ... reachable by Tab" and "nothing is focus-only-via-mouse; ... Tab order matches reading order on every window." Today neither floating panel is Tab/Space operable: both are borderless `.nonactivatingPanel`s in an accessory (LSUIElement) menu-bar app, and such panels do not reliably become the key window, so there is no focus ring and keystrokes never route to them.

UX23 built and reverted a first attempt (reverted in the UX23 close commit; the build-up was in the two prior UX23 commits). What was tried: key-capable `NSPanel` subclasses (`canBecomeKey` returns true), `becomesKeyOnlyIfNeeded = false`, per-control `acceptsFirstResponder` + `keyDown` (Space/Return) + focus rings (a layer border on the layer-only `PromptRecordButton`/`RecordKey`, `drawFocusRingMask` on the `NSButton`s), a `nextKeyView` tab order, and a click-to-focus that forced `makeKey()` + `NSApp.activate(ignoringOtherApps: true)` on empty-area clicks (detected by walking the hit-test ancestry for a first-responder-capable control, so a mouse click on Record/Stop stayed nonactivating). It still showed no focus ring in owner testing, so the root cause is more than "call makeKey": likely the accessory activation policy interacting with the nonactivating-panel key rules. QuickFind (Cmd+K) uses the same makeKey + activate and does work, the difference being that QuickFind is a normal panel the user explicitly invokes while these two appear passively.

What already works and was kept: VoiceOver labels on the prompt's close ("Skip meeting") and chevron ("More options") controls and the HUD's off-the-record toggle, so a screen-reader user can identify them (they just cannot Tab to them). The HUD off-the-record toggle itself (mouse click + right-click menu) is unaffected and shipped with UX23.

The tension to resolve is as much product as AppKit: keyboard focus on macOS is per-key-window and never crosses windows, and making a floating panel the key window in an accessory app appears to require activating the app (a focus steal), which fights the quiet, non-intrusive register these passive panels are built for. So the real question is *when*, if ever, a passively-appearing panel should grab focus. The global hotkeys (toggle / force-stop / off-the-record) remain the keyboard path for the recording flow in the meantime, which is why this is P4: pick it up only if keyboard-driving the passive panels becomes a real need.

Acceptance (when promoted): engaging the prompt or the HUD shows a visible accent focus ring, Tab cycles the controls in reading order, and Space/Return activate the focused control, without the passive appearance of either panel stealing keyboard focus from the meeting app.
