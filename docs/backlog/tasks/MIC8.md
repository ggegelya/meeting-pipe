# MIC8: Spike: UI-independent native mute signals

Band origin: carried from Q4; base spec in q4-final.md, progress notes here. Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**MIC8 (P2, partial): spike UI-independent native mute signals (logs / SDK / OS).** Base spec in [`q4-final.md`](./q4-final.md) (search the id); promoted P3 to P2 and amended 2026-07-12 (see the Assessment review section). **Progress (2026-07-13): analysis + measuring instrument shipped; two of the three candidate signal classes closed from documentation, the log path's per-app verdict is owner-owed.** The spike doc [`docs/spikes/mic8-native-mute-signals.md`](../spikes/mic8-native-mute-signals.md) works the three UI-independent candidate classes and closes two a-priori: **(B) vendor SDK** is a non-starter for the "watch the client the user is already in" model the whole detection architecture rests on (Zoom staff on the developer forum confirm the desktop client exposes no external interaction; the Meeting SDK's `onUserAudioStatusChange` reports mute only inside an app that *joins the meeting through the SDK itself*, the ZoomOSC pattern, not the user's Zoom.app; Teams exposes no local read API either); **(C) OS-level** is a category error (CoreAudio's device-level mute is the hardware input, but a client's in-call mute is a transmit gate that by design keeps the OS mic live, the exact property MIC1's capture-first architecture was built around, so the OS mic-mute cannot stand in for "muted in Teams", and macOS exposes no public read of another app's client mute). That leaves **(A) the client's own local diagnostic log** as the only open thread, and it is undocumented + per-app, so only a live mute/unmute cycle answers "is there a stable line, how fast, does it survive updates." The deliverable, matching CAL1/MIC7/DET2, is an owner-run read-only probe [`daemon/scripts/mic8-native-mute-probe.sh`](../../daemon/scripts/mic8-native-mute-probe.sh) that tails the Zoom/Teams diagnostic logs (Chromium storage subtrees pruned, matches truncated as a privacy guard) and timestamps any mute-token line, so the owner correlates each toggle against a log line. On-disk recon on this Mac frames but cannot answer it: Zoom's diagnostic logs are empty/stale and its verbose logging is opt-in (a signal that needs troubleshooting logging on is itself weak evidence for durability), and Teams' `MSTeams_*.log` is ~405 d stale (recent Teams use looks browser-side). No production code changed, so no regression risk. **Verdict: measure the log path per app; the a-priori lean is a close.** An undocumented log format is no more futureproof than the label scrape it would replace (a vendor log-line change breaks it the same way a UI change breaks the scrape), and it also depends on verbose logging staying on, so MIC8's filed "strongest futureproofing" framing holds only if the line turns out stable *and* default-on. Note the comparison MIC8 was originally filed against is gone: MIC6's AX state-attribute read was **refuted** by the 2026-07-14 live dump (Teams exposes no `AXValue` and no `AXIdentifier` on any element) and closed as unachievable, so the incumbent MIC8 must beat is the localized-description scrape plus MIC10's `AXSystemDialog` structural anchor and the capture-first architecture, which is a lower bar on futureproofing but a much higher one on cost. Owner-owed remainder, the spike's whole point: run `bash daemon/scripts/mic8-native-mute-probe.sh` during a live Zoom and a live Teams call, toggle mute 3-4 times, and read which token (if any) appears and how fast; a stable default-on line promotes MIC8 to a small `LogMuteAdapter` for that app (the MIC7 adapter shape, superseding its `MeetingAXWindowWatcher` read), a flaky / opt-in-only / absent result closes the native-signal bet for that app and leaves the MuteLabels scrape + MIC10's structural anchor + capture-first as the durable answer.
