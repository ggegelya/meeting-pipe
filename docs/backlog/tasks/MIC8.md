# MIC8: Teams `LogMuteAdapter` (native mute signal)

Band origin: carried from Q4; base spec in q4-final.md, progress notes here. Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**MIC8 (P2, partial): build the Teams `LogMuteAdapter`.** Filed as "spike UI-independent native mute signals (logs / SDK / OS)"; base spec in [`q4-final.md`](../q4-final.md) (search the id), promoted P3 to P2 and amended 2026-07-12 (see the Assessment review section). The spike question is **answered**, so the remaining work is a build, not a measurement.

**Spike resolved 2026-07-23: GO for Teams.** Full write-up in [`docs/spikes/mic8-native-mute-signals.md`](../../spikes/mic8-native-mute-signals.md); the measurement is reproducible via [`docs/spikes/mic8_log_correlation.py`](../../spikes/mic8_log_correlation.py) with aggregates committed at [`mic8-log-correlation-results.json`](../../spikes/mic8-log-correlation-results.json).

Two of the three candidate classes stay closed from documentation, unchanged: **(B) vendor SDK** is a non-starter for the "watch the client the user is already in" model (Zoom staff confirm the desktop client exposes no external interaction; the Meeting SDK's `onUserAudioStatusChange` reports mute only inside an app that joins through the SDK itself; Teams exposes no local read API), and **(C) OS-level** is a category error (CoreAudio device mute is the hardware input, while a client's in-call mute is a transmit gate that by design keeps the OS mic live).

**(A) the client's own local log is now measured, and it wins.** Rather than the filed plan of a live instrumented call with 3-4 toggles, it was answered offline against 11 days of real dogfood, which is a strictly stronger read. Teams' SlimCore media stack writes `AirPodsService: Input mute event received: mute|unmute` (plus a `SetMuteState ... cause_id` and an `InputMuteStateChangeNotification` line) on every in-call toggle: 594 transitions across 9 log generations spanning 2026-06-12 to 07-23, identical format throughout, default-on with no troubleshooting toggle involved. Paired against the incumbent AX label scrape from the daemon's own `micgate` events: 61 pairs within 30 s, 59 state agreements, and the log **leads** the AX read on 59 of 61 (p50 50 ms, min 22 ms). Of the four imperfect pairs, three are AX-side defects (a notification that fired 16.8 s after the toggle the log caught in 391 ms; a transition to `unmuted` AX announced when the state had been unmuted for 68 s; a 1.8 s poll lag) and one is an unexplained log gap. It is not candidate C in disguise: across the same window `system_mute_state` was `false` in 34 of 34 observations while the log tracked 60 client-side transitions.

**Why the a-priori lean was wrong, and what the recon missed.** The spike leaned toward a close on the argument that an undocumented log format is no more futureproof than the label scrape it replaces. That argument was made about an unmeasured signal and the measurement beat it: the same vendor broke the AX scrape twice in this project's lifetime (the Teams mini-window incident, MIC9/MIC10) and did not break this line once in 9 generations. The 2026-07-13 recon concluded Teams was not logging because it read `MSTeams_*.log`, the Electron/WebView2 **shell** log; the call media stack writes a different family in the same directory, and its generation `.10` already held 218 mute transitions a month before the spike was written. Two probe defects would have confirmed that false close, and both are fixed and verified in [`daemon/scripts/mic8-native-mute-probe.sh`](../../../daemon/scripts/mic8-native-mute-probe.sh): a flat newest-6-files cut that evicts the only family carrying mute lines (the shell log rotates hourly, SlimCore weekly; observed live, one rotation pushed SlimCore from 5th to 10th newest), and a bare `mute` token that matches the hourly ECS config blob's `hfpOsUnmuteFixEnabled` feature-flag key, which truncated to 200 chars reads exactly like a hit. Note also that the comparison MIC8 was filed against is gone: MIC6's AX state-attribute read was refuted by the 2026-07-14 live dump (Teams exposes no `AXValue` and no `AXIdentifier`) and closed, so the incumbent this beats is the localized `MuteLabels.toml` scrape plus MIC10's `AXSystemDialog` anchor, which the log path also beats on locale-independence by construction.

**Remainder (the build).** A `LogMuteAdapter` conforming to `MicGateAdapter` (alongside `NativeMuteAdapter` / `NoOpMuteAdapter`) that tails the newest `MSTeamsNM_SlimCore_*.log`, parses the transition line, and hands an `AXMuteButtonProbe.Event` to the sink, landing in `MicGate.injectAxMuteEvent` exactly where the AX poll does today. No `PromotionEngine` / scorer change; the mute layer is already an injectable seam. **Owner decision needed before building: supersede the Teams AX read, or fuse with it.** The spike assumed supersede; the measurement supports fuse ("log wins when present, AX carries when absent"), which covers the one case that went against the log, costs one comparison, and does not bet the whole Teams mute oracle on a single undocumented string. Build gotchas, all measured: SlimCore prints **UTC** but suffixes the local offset, so parse the naive reading as UTC or every event is wrong by the offset (and correct-looking on a UTC machine); resolve the file by log family, never by mtime; an absent line degrades to **unknown**, never to unmuted (the `AirPodsService` component name is a misnomer, but this data cannot rule out a dependency on AirPods being connected for output); rotation is per Teams restart, so re-resolve the newest generation rather than holding a file handle. Re-running `mic8_log_correlation.py` after a Teams update is the regression test for the bet.

Zoom stays unmeasured: its logs are empty/stale and its verbose logging is opt-in, and the corpus has recorded zero Zoom calls, so it is effectively moot. The live probe remains the instrument if a Zoom meeting ever happens.
