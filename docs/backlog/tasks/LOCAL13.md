# LOCAL13: Local model refresh bench

Band origin: filed 2026-07-23 (owner ask from the 2026-07-23 local-quality assessment: raise the local floor; cloud parity explicitly not the target). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**LOCAL13 (P2, new): bench newer 4-bit local models against the incumbent and flip `summarization.local_model` on a measured win.** The incumbent is `mlx-community/Qwen2.5-7B-Instruct-4bit` (a 2024-generation model; config.py:122 default, also the owner's configured value). The gap it leaves is documented by the 2026-07-23 side-by-side on stem `20260723-103118` (12.5-minute Teams stand-up, 8,644-char transcript, identical prompt and transcript for both engines): the local pass produced topic-level bullets ("Discussed deployment of hotfix and assigned tasks"), a generic title, and vague actions, where `claude-sonnet-4-6` produced named, specific, checkable output. Model class is the single biggest lever available before any pipeline redesign (LOCAL14), and it is a config flip.

Candidates as of filing (mlx-community 4-bit builds; check for newer at pickup, the space moves): the Qwen3 8B and 14B instruct line (the 2507 refresh), Gemma-3-12B-it QAT, Phi-4-14B. Constraints from prior owner-Mac measurements (the AI2/LOCAL12 protocol): the 14B class OOM-hangs this Mac at long context, so 14B candidates run at 4-8K context only; every run goes under `caffeinate` with a free-RAM watchdog (abort below ~10% free) and results written incrementally, never buffered to the end. A 12-minute meeting is ~2.5K tokens, well inside every candidate's window, so the bench measures quality, not context reach.

Probe legs first (each a few minutes, results recorded in the bench doc): (1) the installed `mlx_lm` version loads each candidate arch at all (`mlx_lm.server --model X` + one `/v1/models` health call via the existing `build_server_command` path); (2) whether the server honors the `response_format: json_schema` hint `_chat_completion` already sends (summarize_local.py:684-714) per candidate, since constrained decoding materially changes JSON reliability and LOCAL14 wants this answer too; (3) tokenizer/template quirks that break the schema-as-text discipline (a thinking-mode default that prepends reasoning text would today land in the 3-layer extractor; note it, do not fix it here).

The bench: `mp dogfood` over a fixed set of 4-6 representative transcripts (EN and UK mix, short and mid-length, including `20260723-103118`), incumbent vs each candidate, same prompts, hand-scored per the existing scorecards (`actions_capture`, `decisions_capture`, `hallucination_rate`) plus recorded wall-clock per summarize and peak RSS. Pre-register the ship thresholds in the bench doc BEFORE scoring (the /spike discipline): flip when a candidate beats the incumbent on both capture scores without a hallucination regression, stays within the RAM guard, and lands under an owner-chosen latency ceiling; otherwise keep the incumbent and say why. Owner-owed: the runs and the scoring (needs the real Mac and real judgment); the session prepares the harness invocations and the doc skeleton.

Explicitly not: no prompt changes and no pipeline-shape changes in this bench (that is LOCAL14; mixing levers destroys attribution); no LoRA (LOCAL9 compares base vs adapter separately); no Apple Intelligence in the comparison (different backend class, LOCAL8/EGRESS1 territory); no automatic model download during NDA meetings (the egress guard's prefetch rule stands, `mp prefetch-model` outside a zero-egress run).

Acceptance: a committed bench report (docs/spikes/ or the dogfood runs dir) with the pre-registered thresholds visible above the scores, per-model scorecards, latency and peak-RSS table, and a named verdict; `summarization.local_model` flipped in the owner's config on a win (and `mp doctor` + `prefetch-model` verified against the new repo id) or explicitly kept with the reason recorded; the probe-leg answers (arch support, response_format honoring, template quirks) recorded for LOCAL14's pickup. No repo code changes expected; if a probe reveals a one-line compatibility fix is needed to run a candidate at all, it ships separately with its own test.
