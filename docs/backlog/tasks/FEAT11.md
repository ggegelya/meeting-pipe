# FEAT11: In-app context instruments (glossary editor, voice-resolution surface, context nudges)

Band origin: filed 2026-07-23 (owner ask, 2026-07-23 assessment session; owner's framing verbatim: "I want the application to provide me instruments so I can do things", nothing done at any level other than the app itself). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**FEAT11 (P2, new): the three cheapest summary-quality levers exist only as hand-edited config or are invisible; give each an in-app instrument.** These levers move BOTH backends and disproportionately help the small local model, because they turn facts the model must infer (who is speaking, what "PS for Venusaur" was supposed to be, what this team's vocabulary means) into facts written in the transcript. The owner's operating rule is that no lever should require editing a TOML or running a CLI.

Context, current state per lever (verified 2026-07-23). (1) Glossary (ASR1): fully built pipeline-side (`mp.glossary`, deterministic normalization at finalize, global `[terms]` + per-workflow `[workflow."<name>".terms]`, optional `[fuzzy]`), but it has zero daemon UI (no hit for "glossary" anywhere in daemon/Sources) and zero presence on the owner's Mac (`~/.config/meeting-pipe/glossary.toml` does not exist), so ASR garbles like "Resolve PS for Venusaur ticket" (stem `20260723-103118`, local summary) reach both backends raw. (2) Voice naming: the instruments exist (TranscriptTab in-app naming enrolls the voiceprint, FEAT3; Preferences People pane manages the roster) and the owner's roster.json is populated (45 KB, last written 2026-07-16), yet that meeting still surfaced `THEM-B`/`THEM-C` and NO surface says whether roster matching ran, matched, or missed, so the owner cannot tell "not enrolled" from "enrolled but unmatched" without reading sidecars. (3) Workflow context: the editor exists (`WorkflowsView.swift:375-393`, Context prompt group with a hint) but both of the owner's NDA "Client work" workflows carry an empty `context_prompt`, so NDA meetings summarize with no team context at all; nothing nudges toward what good context contains.

Scope. (a) Glossary editor: a Vocabulary surface (Preferences, Pipeline pane or a sibling) that lists wrong -> right term pairs, a global table plus an optional per-workflow section, reading and writing `~/.config/meeting-pipe/glossary.toml` with the ConfigStore TOML round-trip idiom (unknown keys and the `[fuzzy]` section survive untouched; `glossary.example.toml` stays the documented reference). Inline hint: terms apply at finalize, so already-processed meetings pick them up on Reprocess (and in bulk only via ASR3). A small "add to vocabulary" affordance from the Transcript tab's existing text-correction flow (PIPE9) may pre-fill the pair from an edit; keep it a pre-fill, not an auto-write. (b) Voice-resolution surface: in the meeting detail, show the cast as resolved names vs unresolved cluster labels (the data is already in the transcript + speaker overlay), with a one-click path into the existing TranscriptTab naming flow for each unresolved voice; surface the roster-match outcome for the meeting (matched n of m voices) from the pipeline's existing events or, if no durable record exists, name that gap at pickup and add the smallest sidecar field or event that carries it. (c) Context nudges: an empty `context_prompt` renders placeholder guidance naming what actually helps (people and roles, product and ticket vocabulary, meeting cadence), and the workflow list shows a quiet hint chip on workflows with empty context; no new storage, no auto-fill.

Explicitly not: cross-meeting frequency mining ("this unnamed voice appeared in 6 meetings", needs embedding correlation across the library; park until the per-meeting surface exists); auto-suggested glossary terms from the corrections corpus (real idea, separate filing once the editor exists); glossary application at display time (finalize-only is the ASR1 contract and keeps transcripts deterministic); editing roster embeddings or thresholds in-app (DIAR2 owns matching quality); any change to what `mp.glossary` or `mp.roster` compute.

Acceptance. Daemon tests pin: glossary TOML round-trip (unknown keys + `[fuzzy]` survive; per-workflow table wins on a clash exactly as `mp.glossary` resolves it, pinned against the Python reader with a shared fixture in the CI4 idiom if practical, otherwise a mirrored unit pair); the voice-resolution presentation model (resolved vs unresolved rows, deep-link target); the empty-context hint logic. Suites + `scripts/truth_fences.py config` green (glossary keys stay documented). Owner eyeball: define "Venusaur" -> the intended term in-app, Reprocess `20260723-103118`, and the garble is gone from transcript and summary; name one `THEM-*` voice from the detail surface and see the cast row flip to the name.
