# LOCAL9: Corrections Phase 3: local LoRA fine-tune

Band origin: feature review 2026-07 (owner approved 2026-07-05). Status and priority live in this task's ToC row in [meetingpipe-q6-backlog.md](../meetingpipe-q6-backlog.md).

**LOCAL9 (P2): corrections Phase 3, local LoRA fine-tune.** `mp corrections-stats` advertises a Phase 3 readiness check (~20 corrections over ~200 minutes) that no backlog task owns; make it real or stop advertising it. Scope: an owner-invoked `mp train-adapter` that fine-tunes a LoRA on the standard 7B via the MLX LoRA path, building prompt/completion pairs from the corrections corpus (original vs corrected summaries); evaluation is `mp dogfood` A/B (base vs adapter) on held-out meetings, hand-graded like the engine comparison; the adapter ships only through an explicit config opt-in (`summarization.local_adapter_path`), never silently applied. Fully on-device, no egress; training is a command the owner runs, no background training. An honest terminal outcome is allowed: if the A/B shows no capture win, close as adapter-not-worth-it, keep the corpus as a prompt-context asset (few-shot exemplars are the cheaper fallback lever), and fix the README wording. Gated on the corpus bar, which is owner-owed grading volume. Classification note (2026-07-10): `mp train-adapter` is owner-dev tooling under the de-CLI standing rule, CLI-only by design, like dogfood. Acceptance: with the bar met, a training run produces a loadable adapter; the dogfood scorecard decides adoption; README's corrections section describes the real flow either way.

**Progress (2026-07-23): the blocker was the corpus source, not the tooling.** Measured first: the corrections dir has held **0 records** since it was created on 2026-05-18, while the recordings dir holds **169 run sidecars, all with transcript and summary on disk, 2,422,719 transcript chars** (12x the 200k bar), 167 of them summarized by `claude-sonnet-4-6`. So the spec's only training source was empty for two months next to a ready-made distillation corpus. This pass added `--source {corrections,runs,both}` (default unchanged): `runs` builds pairs from the run sidecars (transcript -> the summary that run published), skipping local-backend runs since a model learns nothing from its own output; `both` puts corrections first and withholds their stems from the runs fill. Measured end to end on the real corpus: 167 pairs, train=135 / valid=16 / test=16.

Three defects surfaced while validating against real data, each of which would have made a training run worthless without failing loudly. (1) **Train/inference prompt mismatch.** `_base_instruction()` hand-substituted two of the master prompt's four placeholders, so every training prompt carried a literal `{extra_sections_directive}`, said "(no team context configured)" where the server sends the real team context, wrapped the transcript in `<transcript>` tags the server never uses, and omitted the schema block entirely. Fixed by building the pair from the shipping helpers: `summarize._load_system_prompt` plus `summarize_local.augment_with_schema` / `.compose_user_message`, the last two promoted from `LocalSummaryClient` methods to module functions so there is one copy. (2) **Wrong dataset format.** `{prompt, completion}` has no system role, so the system turn was being folded into the user turn; now emits `{"messages": [...]}` (mlx-lm's `ChatDataset`), the same three turns the server exchanges. (3) **Silent truncation.** `mlx_lm.lora` defaults to `--max-seq-length 2048` and truncates the tail, which is where the summary lives; 106 of 135 real pairs exceeded it, so most examples would have taught the model nothing at all. Now an explicit `--max-seq-length` (default 4096) with over-long meetings dropped and counted in the log, never truncated, plus `--mask-prompt` (loss on the summary, not the transcript) and `--grad-checkpoint`.

The A/B is now actually held-out: training writes `split.json` beside the adapter recording which stems went to train/valid/test, and `mp dogfood --adapter` refuses a transcript the adapter trained on, naming a held-out one instead (`--allow-trained` overrides; an adapter with no manifest warns rather than blocks, since it cannot be verified either way). Verified: 913 pipeline tests green, ruff and pyright clean, CI5 fences pass, and `mp train-adapter --source runs --dry-run` exercised against the owner's real 169 meetings. **Owner-owed remainder** (unchanged in kind, now actually runnable): run `mp train-adapter --source runs --adapter-path <dir>` on the M1 Pro, then `mp dogfood --adapter <dir> <held-out transcript>`, hand-grade the scorecard, and set `summarization.local_adapter_path` only if it wins. An honest negative result stays a valid terminal outcome. Note that LOCAL11 (warm-server identity) is a real precondition for trusting the A/B: a warm base-model server would otherwise serve base weights on both sides.

**Both preconditions are now resolved; the A/B is cleared to run (2026-07-23).** LOCAL11 shipped (commit 65ba629) and LOCAL15 shipped in the same session that raised it, so a configured adapter genuinely serves and a warm base-model server no longer silently answers both sides of the comparison. Verified end to end against a real `mlx_lm.server`: a summarize run carrying the adapter returns a summary, and a `local_adapter_path` that does not exist stops the run naming the path instead of quietly producing a base-model summary. The remaining work is the owner's training run, hand-grading, and the adoption call. The original blocker, kept for the trail:

~~**Do not run the A/B until LOCAL15 lands (filed 2026-07-23).**~~ A second, more fundamental precondition surfaced while shipping LOCAL11: mlx-lm 0.31.3 silently drops the `--adapter-path` flag this task's serving seam passes (`ModelProvider.load` keys its `_adapter_map` lookup by the already-resolved model path while the map is keyed `"default_model"`), so `summarization.local_adapter_path` is a no-op and the base model serves regardless. Verified by code read, empirically against a stubbed `ModelProvider`, and upstream (ml-explore/mlx-lm#1248 open since 2026-05-06, fix PR #1249 unmerged). Running the A/B before that is fixed compares the base model against itself, and the false negative would read as an honest "the adapter did not help" and wrongly close this task after real training and grading time was spent. Full evidence and scope in [LOCAL15](./LOCAL15.md).

**Progress (2026-07-11): tooling shipped, adoption owner-owed.** The full tooling is in: `train_adapter.py` (`mp train-adapter`) reads the summary-grading corpus, builds `{prompt, completion}` pairs (the base summarization prompt + transcript, from `mp.prompts/meeting_summary.md`, mapped to the owner's corrected summary JSON), splits train/valid/test, and shells out to `mlx_lm.lora --train` (subprocess, matching `build_server_command`'s precedent, so the module stays importable on Linux CI); it refuses below the `corrections-stats` readiness bar unless `--force`, and runs local-only via `entry.prepare(secrets=False)`. `summarization.local_adapter_path` (config, empty default) is the opt-in, threaded through `summarize_local.build_server_command` (+`--adapter-path`) and both `LocalSummaryClient` construction sites plus the `mp serve-local` warm path, so a configured adapter is actually served and never applied when unset. `mp dogfood --adapter <dir>` adds the base-vs-adapter A/B (local-only, no cloud baseline, no zero-egress refusal). README's "Improving local quality" section, `config.example.toml`, and the ARCHITECTURE subcommand table describe the real flow. Verified: pipeline 673 green (ruff + pyright clean), incl. pair-building / split / command-construction / `--adapter-path`-when-set / dogfood-adapter-mode-skips-cloud tests; `mp train-adapter --help` and the readiness gate (0/20 on this Mac) confirmed by hand. The two owner-owed remainders keep this partial: the **corpus is empty on this Mac** (grading volume is owner-owed), and the **A/B adoption decision** (run `mp train-adapter` then `mp dogfood --adapter`, hand-grade, set `local_adapter_path` only if it wins) needs a real corpus + Apple-Silicon Mac, not exercisable here. An honest negative result stays a valid terminal outcome.
