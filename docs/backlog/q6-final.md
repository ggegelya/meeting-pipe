**[DONE] STOR4 (P2): scheduled automatic backups.** Shipped 2026-07-23. Band origin: assessment review 2026-07-12. Original scope in the ToC row above. The corpus is irreplaceable (the 2026-07-05 review's own framing), yet `mp backup` ran only when the owner clicked "Back up now" or remembered the CLI, and doctor merely reported how stale that was: observing the staleness is not fixing it.

**The agent.** A new `BackupSchedulerService` installs `~/Library/LaunchAgents/com.meetingpipe.backup.plist`, a `StartCalendarInterval` running `mp backup <destination>`, and `launchctl bootstrap`/`bootout`s it, driven by an "Automatic backup" toggle in Preferences ▸ Storage beside STOR3's existing destination picker + "Back up now" button. The destination is baked into the plist's `ProgramArguments`, so changing it in Preferences rewrites the agent rather than silently keeping the old folder; with no destination the agent is removed however the toggle reads, since an agent invoked without a directory would fail every night into a log nobody reads. The STOR3 last-backup age line is the run's visible outcome, unchanged and already in the right place.

**Deviation 1 (shared mechanism, not a copy).** The spec said "copy the AI4 launch-agent mechanism exactly". Copied literally that is ~60 lines of plist writing plus the launchctl dance duplicated, and this repo has a documented precedent for what that costs: `publish_fs`'s module-private copy of `_render_summary_md` drifted from the original until PIPE7 deleted it. So the mechanism was **lifted** into a shared `LaunchAgentScheduler` (plist body, install/remove, `launchctl`), with `DigestSchedulerService` and `BackupSchedulerService` reduced to what is actually theirs: label, `mp` subcommand, events. Two real callers, no speculative generality. The digest's public API and its `digest_schedule_installed` / `_removed` events are unchanged.

**Deviation 2 (daily, not weekly).** The digest is weekly; a backup defaults to **daily**. A weekly cadence on an irreplaceable corpus means up to seven days of new meetings exist on exactly one disk, which is the risk STOR4 was filed against. The frequency is a picker (Daily / Weekly); daily omits launchd's `Weekday` key entirely, which is how launchd expresses "every day", and the Day picker greys out. Defaults: daily at 21:00, late enough that the day's meetings are in, early enough that the Mac is usually awake. A Mac asleep at that hour backs up on wake.

**Deviation 3 (an unmounted-drive refusal in `mp backup`).** Not in the spec, and a real hazard that unattended scheduling *creates*: `mkdir(parents=True)` cannot tell "the drive is unplugged" from "the folder is new", so a scheduled backup to `/Volumes/Backups` with the drive away would create that path on the startup disk, write gigabytes into it, stamp the marker, and report success. Doctor and Preferences would then show a *fresh* backup that is on the wrong disk, and the real drive would mount at `/Volumes/Backups 1` next time it was plugged in. That defeats this task's own acceptance ("last-backup age stays fresh") by making the age line lie. `backup.check_destination` refuses exactly that signature (a missing volume root under `/Volumes`, nothing else) before anything is created or stamped, exiting 1 with a message naming the fix; a mounted volume and every path elsewhere behave as before.

**[DONE] LOCAL11 (P2): warm-server identity verification.** Shipped 2026-07-23. Band origin: assessment review 2026-07-12. Original scope in the ToC row above. The filed problem was real and reproduced: `_is_healthy` checked HTTP 200 on `/v1/models` and nothing else, so a warm `mlx_lm.server` kept serving whatever it was started with after a `local_model` / `local_adapter_path` flip, `_identify_backend` recorded the *configured* model in `<stem>.run.json`, and a LOCAL9 adapter A/B could not distinguish "the adapter did not help" from "the adapter never served".

**One deviation from the filed scope, forced by measurement.** The spec said "compare the `/v1/models` payload against the configured identity". That endpoint cannot answer the question. mlx-lm 0.31.3's `handle_models_request` scans the HuggingFace cache and lists *every* downloaded MLX model, appending `--model` only when it is a filesystem path, so a repo-id server is indistinguishable from any other cached repo; `/v1/chat/completions` is no better, echoing back the `model` the caller itself sent (`self.requested_model`). So identity is read from the **server's own argv**: `lsof` on the port, `ps` on that pid, parse the `--model` / `--adapter-path` flags `build_server_command` emitted. That is strictly better than the filed route in the way that matters: it covers a server *we did not spawn*, including the daemon's warm `mp serve-local` (which `execvpe`s, so its argv is the server's) and one a previous `mp` left behind. The LOCAL10 marker gained `adapter_path` and serves as the fallback when `lsof` is unavailable. The `lsof`-on-the-listening-port shape is the one LOCAL12's bench already uses.

**As built.** `local_server.served_identity(port)` returns a `ServedIdentity` (pid, model, adapter, source `argv`/`marker`) with a pure `parse_identity` underneath, reading each flag value to end-of-string-or-next-` --` so an adapter path containing a space (`~/Library/Application Support/...`) is not truncated. `LocalSummaryClient._reuse_is_safe` runs it before reusing a warm server, and `terminate_server` (extracted out of `reap_orphan`, which now calls it) does the killing. Three rules, each unit-pinned: **unknown is not mismatch** (an unparseable `ps` line, nothing listening, or a non-mlx process on the port all reuse, because respawning on ignorance costs a multi-GB reload); **only kill what you spawned** (this process, or a marker naming that exact pid; never the daemon's warm server, which would race its supervisor); **never launder the mismatch** (when the run proceeds anyway, `model` / `adapter_path` report what *served*).

The daemon owns the other half, because Python rightly refuses to kill what it did not spawn: `LocalModelPreloader` gained an `Identity` + a pure `decide(running:desired:)` and a `refresh(...)` wired to `ConfigStore.didPersist` (already debounced 500 ms), so a model or adapter change restarts the warm server and an unrelated save is a no-op. The restart is a delayed main-queue hop, not a `waitUntilExit()`: blocking main on a subprocess is the freeze class the audio and AX paths already paid for, and if the rebind loses the port race the failure mode is a cold first summary, never wrong weights, since `LocalSummaryClient` then spawns its own correctly-identified server. `ConfigStore.summarizationLocalAdapterPath` is a read-only computed read off `rawDocument`, following `outputSinks` rather than becoming a persisted knob with no Preferences control (which is what the CI5 dead-knob fence exists to catch).

Attribution was fixed on the `auto` ladder too, which the spec did not name: `_AutoFallbackClient` was reporting `cfg.summarization.local_model` after a cloud-to-local fallback, the same mis-attribution one rung further in. It now reads the served pair off the client before the `with` block closes it, through a deliberately tolerant `_served_local_identity` (attribution metadata read *after* a summary already succeeded must never cost the summary, the same rule `orchestrate` applies to a failed sidecar write). `write_run_sidecar` gained `adapter_path`, always present and `""` for every backend but a local one serving an adapter, so a reader never has to distinguish "no adapter" from "written by an older `mp`".

**UI affordance:** none built, none needed. The doctor probe surfaces verbatim in Preferences → Integrations → Run doctor… (`DoctorRunner` spawns `mp doctor` and streams its whole output into the sheet); the respawn and the daemon restart are automatic.

**Verified:** 937 pytest, 1433 swift test, ruff, pyright, CI5 truth fences, all green. 24 new Python tests (13 over the identity layer, 5 over client reuse/respawn, 5 over the doctor probe, 1 over the sidecar) and 6 new Swift tests over the restart decision. Docs: CONVENTIONS (marker schema gains `adapter_path`; a new "served identity is read from argv" section with the three rules and the mlx-lm finding), ARCHITECTURE (`local_server.py` entry rewritten; `LocalModelPreloader.swift` added to the daemon file list), README (a "changed the model and summaries did not change" troubleshooting entry), `daemon/CLAUDE.md` (pipeline-subprocess index row).

**Known limitation, stated rather than papered over:** argv is the *spawn contract*, not a readback of the loaded weights. It catches a server started for a different model or adapter; it cannot catch a server that failed to apply an adapter it was correctly handed. That distinction is load-bearing here, because mlx-lm 0.31.3 appears to drop `--adapter-path` on its own: `ModelProvider.load` looks up `self._adapter_map` with the *already-mapped* model path while the map is keyed `"default_model"`, so `load_default()` resolves the adapter to `None`. If that reading holds, LOCAL9's `--adapter-path` seam serves the base model regardless of anything meeting-pipe does, and the working route would be the per-request `adapters` body field. Left unfixed deliberately: it is an upstream defect in LOCAL9's serving seam, not LOCAL11's identity question, and changing how the adapter is requested wants the owner's call plus a measured A/B. Worth filing before the LOCAL9 remainder is attempted, or that A/B will measure the base model against itself.

The DV pivot's re-open condition is now structurally met: FEAT3-ROSTER and FEAT3-MANAGE shipped, so named, stable, cross-meeting speaker identities exist. Scope: a Facts-style INSIGHTS projection ("People"): one row per enrolled person showing their meetings, the open actions they own (the summaries already carry owners), and last-seen; rows link back to meetings (the Quick Find hook), rename flows through the existing roster rename. Read-only projection over existing sidecars, no schema change; local-only per the AI/DV rule. Gate: needs a minimally populated roster, which is owner-owed enrollment volume; build after real usage proves a handful of stable identities. Acceptance: selecting a person lists their meetings and open actions; a roster rename reflects everywhere; the empty-roster state points at how to enroll.

**Shipped 2026-07-24.** The gate cleared before a line was written: the owner's roster held 4 enrolled people, and a dry run of the derivation over the real 193-summary library gave Heorhii 27 meetings / 66 open actions, Ivan 18 / 24, Maharsh 6 / 3, Rana 3 / 4. Not a rail that would render empty on day one.

**Where identity comes from.** The roster is the spine, not the transcript. A raw diarization label is not a person (`MeetingCast` documents why), and only an enrolled voice carries a name across meetings, so the rail has one row per `roster.json` entry and none for an unenrolled owner string. That is a real exclusion in this library, and deliberately so: the top action owners include `1`, `2`, `3`, `speaker_0`, and `THEM-A`. An identity you cannot rename or merge is not an identity, and a rail full of them would be a rail nobody could maintain.

**Where membership comes from.** Two sources, unioned, because each alone under-reports. (a) The summary's `attendees`, resolved through `<stem>.speaker_labels.json` so an in-app naming counts even when the summary predates it (and a name the overlay introduced counts even when the summary never listed it, the `MeetingCast` "New person" case). (b) Ownership of one of the meeting's open actions, because the summarizer names owners the diarizer never labelled: Heorhii is an attendee in 27 summaries but an action owner in 66, so attendance alone would have hidden meetings whose actions the rail was already showing. Matching is trimmed + case-insensitive exact. Nothing looser: first-token or substring matching would fold two different Ivans into one row, which is the wrong-name failure the roster's own two-gate accept rule exists to avoid.

**Why it does not read transcripts.** The projection re-derives on every `MeetingStore.revision` bump. The library's `<stem>.json` transcripts total ~80 MB against ~1 MB of `<stem>.summary.json`, and the summary's `attendees` is the same speaker set the transcript produced, so the cheap source is also the correct one. `<stem>.embeddings.json` was considered and rejected: it exists for 27 of 175 meetings.

**The one deviation from "read-only", and why it was necessary.** The spec framed this as a read-only projection whose rename "flows through the existing roster rename", with the acceptance line "a roster rename reflects everywhere". Those two do not compose: `mp roster rename` renames the entry in `roster.json` and nothing else, so summaries and transcripts keep whatever name they were written with, and a rename would have emptied the renamed person's history in the very rail that lists it. Shipping the rail without addressing that would have made a pre-existing FEAT3-MANAGE defect newly visible while leaving it broken. So the rename **carries**: after the CLI call lands, the new name is written into the reversible `<stem>.speaker_labels.json` overlay of every meeting the person appears in. Three shapes are covered, because a person reaches a meeting three ways: the pipeline baked the roster name into `<stem>.json` at finalize (no overlay entry, so map the raw label), the name came from an in-app naming (rewrite the overlay value), or a regenerate did both. `<stem>.json` is never rewritten (FEAT3-UNDO's invariant) and identity mappings are pruned, so renaming back collapses what the carry added rather than leaving "A displays as B" behind. A failed carry is not a failed rename: the roster entry has already moved, so the per-meeting failure is logged and the caller is told the rename succeeded.

**One rename path, two surfaces.** `RosterRename` owns the prompt, the CLI call, the carry, and the events; the People rail and Preferences ▸ Pipeline ▸ People both call it. Preferences previously had its own copy of the prompt and the rename, which is exactly how the two would have drifted.

Touched: `LibraryScope` / `LibrarySidebar` / `LibraryWindow` (the `.people` scope, its INSIGHTS row placed second, its center-column branch), `Coordinator+Automation` (`meetingpipe://library?scope=people`), `SpeakerLabelStore` (a `replace(overlay:)` single write, so the carry does not pay N read-modify-writes), `PreferencesView+Pipeline` (onto the shared path; its own `promptForName` deleted).

The empty state names the only way in (name a speaker from a transcript), since Preferences and Facts are both dead ends when nobody is enrolled.
