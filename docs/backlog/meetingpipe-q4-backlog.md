Q4 carries forward everything Q3 left open (not started, "done but ..." with an unmet bar, and items closed at low confidence), plus the findings from the Q4 full-scale review (architecture, security, performance, design) and the workflow-editor review, plus the mic-capture and mute-detection rearchitecture from the Q4 mute-capture audit (the `TECH-MIC` band).

## How to pick up a task (delegation)

Mechanics are codified in `/tech-task <ID>` (read the task here, read the orientation docs, implement, verify, one commit on `main`, do not push). This section is only the soft context that command does not carry:

- User locale: 99% English / Ukrainian. Do not default examples to German. `uk` is in `MuteLabels.toml`; verify any new `uk` labels with the user.
- The user is vibe-coding: high-level explanations, does not read code. Lean on the ARCHITECTURE.md diagrams.
- No em-dashes in any output (CI fails on a U+2014 in an added line; a whole-tree ban under `daemon/Sources` and `daemon/Resources`). Use hyphens, commas, or rewrite. Do not hardcode a personal name or email; commit with the repository's configured git identity. Do not push without permission.
- MicGate runtime knobs live in Preferences > Recording > Microphone and Preferences > Prompt > Stop conditions.
- Binding constraint (from CLAUDE.md): the primary user is the author, selling is tertiary. Technical excellence in core functionality comes first. The one promotion this quarter: GitHub repo presence and the app's visual identity are in scope (TECH-REPO1/REPO2), for contributor visibility. The full launch BRAND band stays deferred.

## Priority bands

- **P0**: correctness or a broken core promise; do first.
- **P1**: meaningful, not blocking.
- **P2**: polish and power-user payoff.
- **P3**: deferred; listed so the trail stays readable.

## Task specs

### Mic capture and mute detection (rearchitecture, Q4 mute-capture audit)

Origin: a two-auditor review of the recurring "talking unmuted, mic recorded nothing" failure, cross-checked against the live `events.jsonl` (21,252 micgate events over 19 days, every headline count re-verified) and the source, then pressure-tested by an adversarial panel. Both audits' diagnosis is confirmed: the mic gate is fail-closed (`passesLiveAudio` is true only for `.hot`, [MicGateVerdict.swift:31](../../daemon/Sources/MeetingPipeCore/MicGate/MicGateVerdict.swift)), reaching `.hot` requires not-muted AND voice-energy with no confidently-unmuted floor ([MicGate.swift:165](../../daemon/Sources/MeetingPipeCore/MicGate/MicGate.swift)), and a stale AX `.muted` latches for the whole call because the probe never re-arms its element ([AXMuteButtonProbe.swift:162](../../daemon/Sources/MeetingPipeCore/MicGate/Probes/AXMuteButtonProbe.swift)) and the blind-clear rescue is structurally unreachable (it fired 0 times in 19 days, [MeetingAXWindowWatcher.swift:167](../../daemon/Sources/MeetingPipe/MeetingAXWindowWatcher.swift)). The decisive finding the audits only half-stated, and a macOS-API sweep then confirmed: there is no general, cross-tool, real-time API to read whether another app's mic is muted (Core Audio process objects expose running-input but no mute selector; AUVoiceIO muted-talker detection is first-party only; ScreenCaptureKit cannot read mute; CallKit only sees your own app's calls; AX label-scraping is the only per-app oracle and it is absent for every browser meeting). So a real-time destructive capture gate that depends on the mute read can never be reliable across tools, and that dependency, not any single AX bug, is the architectural root cause.

Industry check (how the mature notetakers do it): Notion AI Meeting Notes and Granola both capture mic + system audio at the OS layer and do no per-app mute detection at all; cross-tool generality comes precisely from not depending on per-app mute state. Notion detects a meeting by "a process actively using the microphone (e.g. Zoom)" plus a calendar match and does not read mic content, gates on an explicit consent click, captures system+mic on desktop, stores a temporary local audio copy deleted after successful processing or within 24 h (persistent local storage is off by default, with an Enterprise auto-deletion schedule), and sends audio to transcription sub-processors that do not store it. Granola captures device audio, transcribes in real time, deletes the audio immediately, and keeps only the transcript, across Zoom / Meet / Teams / Slack / Webex via the one generic capture path. Neither honours in-app mute; privacy is managed by retention (ephemeral audio, transcript-only), not by gating. meeting-pipe's capture and detection layers are already industry-aligned (ScreenCaptureKit system audio + mic; process-audio + window detection); its only divergence is the real-time destructive MicGate, which no competitor has and which is the bug source.

Direction (ratified in TECH-MIC1): keep lossless OS-level capture as the load-bearing layer and never gate capture on a real-time mute read; manage default-mode privacy by retention the way Notion/Granola do; demote the mute signal to an optional, per-tool, stable-signal layer that drives an offline redaction for the strict/regulated differentiator (MIC5) and, where no audio at rest is required, a hardened real-time gate (MIC6). The user has approved tool-scoped approaches (Teams, Zoom, Slack, Google, ...) provided they are stable and futureproof against UI/UX changes, so the oracle tasks favour durable signals (a button's state attribute over its localized label, platform DOM/extension state, local logs) over the current localized-title scrape the incident proved fragile. Sequencing: MIC1 decides; MIC2/MIC3 are correctness fixes that are right under any architecture (do them regardless); MIC4 is the cure; MIC5/MIC6/MIC7/MIC8 are the optional privacy layer and the per-tool oracles.

### Meeting-end detection (Q4 end-detection audit)

Findings from the Q4 end-detection audit: a two-part review of how we decide a meeting ended versus the mature local bystander notetakers (Granola, Notion, Cluely, Fireflies, Otter, Fathom), cross-checked against 19.8 days of live `events.jsonl` (57 sessions, 100% Teams + manual tooling) and the adversarially-verified macOS-audio API surface. The ADR 0008 verdict-fusion architecture is sound and is the same shape the best competitors use (multi-signal lifecycle fusion, capture decoupled from detection), so this band hardens the signals and the backstop; it does not reshape the fusion, and a rewrite is not warranted. Three verified defects drive almost all of the misses: (1) the process mic-release signal, enabled for Teams/Zoom and the industry's primary ad-hoc end trigger, has produced 0 successful reads in 19.8 days (13,407 `process_audio_unresolved`, the PID-to-AudioObject translation returns object 0), so it is non-functional rather than under-used; (2) signals never corroborate, all 40 `lifecycle.ended` carried an empty `confirmed_by`, so every end rides one signal plus the full 2.0s debounce and ~43% of substantial Teams meetings were missed and stopped by hand; (3) the silence backstop is RMS on both channels at -50 dBFS, defeated by any ambient audio, and barely fires (5-min twice, 8-min never in 19.8 days). Calendar scheduled-end was evaluated and is deliberately NOT in this band: the user confirmed meetings are not reliably calendar-bound, which is ADR 0011's own low-value driver, so ADR 0011 stands and no EventKit permission is added. Sequencing: END1 is the linchpin spike, do it first; END2's device-idle and window legs are independent and can land in parallel while END1's mic-release leg waits on it; END3 and END4 are independent of both; END5 is deferred until a browser meeting is actually recorded.

### 2026-06-10 full audit (implementation, security, UX)

Origin: a 53-agent full audit (10 domain finders, adversarial refuter per finding, competitor scan across Granola/Notion/Fathom/Otter/Krisp/MacWhisper/Superwhisper/screenpipe), run while the END band landed in a parallel session; every end-detection finding was re-verified by hand against HEAD `35fa406`. The audit's headline: the recurring detection pain is now mostly a prompt/suppression-lifecycle problem (start side), and the biggest untracked risks are post-capture data destruction (REC1), crash-path privacy egress (REC2), and systematically silent failures.

**TECH-START1 (P0 new): un-wedge the prompt/suppression lifecycle (AUD-1, AUD-2).** Two coupled defects: (a) in `.starting` (prompt up / after Skip), the FIRST uncorroborated `.ended` from any signal sends PromotionEngine to terminal `.ended` (`handleStarting`, `PromotionEngine.swift:169-178`; the END2 `requiresCorroboration` guard covers only `tick()`), while `PromptEndPolicy` holds `.prompting/.suppressed`; the engine then absorbs everything, the genuine end never fires, and discovery (gated on `.idle`) is dead until restart. (b) Suppression is global and end-gated: skipping one Teams meeting suppresses every app's detection for as long as any Teams window keeps the loose "microsoft teams" SCK token alive. Fix: extend `requiresCorroboration` to `handleStarting`; reset/re-engage the engine whenever `PromptEndPolicy` withholds an end (plus a belt-and-braces guard in `engage()`: reset the engine if it is still terminally `.ended` from a prior session); scope suppression per-bundle with a TTL or discovery-driven re-check; require call-shaped titles for SCK liveness (keep the loose token for discovery only). Acceptance: skip a Teams prompt, end the call, start a second meeting (same and different app) within 10 min; both prompt. Confidence 88/90.

**TECH-START2 (P0 new): make plain browser-tab meetings discoverable (AUD-3).** Browser candidates are admitted only when a `meeting_apps.toml` URL fragment appears in the AX window title; browsers put page titles there, never URLs, so a regular-tab Meet/Teams-web/Zoom-web meeting can never prompt (`MeetingSourceScanner.swift:93-94,204-216`; no `kAXURLAttribute` read exists in the daemon). Fix: match browser windows against the page-title patterns the lifecycle adapters already own (`MeetingTitlePatterns`), exactly as the scanner does for PWAs; keep URL fragments as a bonus signal. Acceptance: a Meet call in a plain Chrome/Safari tab prompts within the scan interval. Confidence 88. Unblocks END5/MIC7 (both assume discovery works).

**TECH-START3 (P1 new): require an in-call corroborator for name-only PWA admissions (AUD-4).** An idle meeting-named PWA prompts on `localizedName` alone (synthetic titleMatch, `.browser` exempt from the lone-candidate corroboration gate, `isMeetingPWA` bypasses the SCK liveness closure), then a timeout-skip triggers the START1 suppression. Fix: pass the real `titleMatched`, extend `hasCorroboratingSignal` to name-only browser admissions, drop the `isMeetingPWA` SCK bypass. Confidence 93 (events confirm it fired on this machine).

**TECH-REC1 (P0 new): never delete capture intermediates on an unverified merge (AUD-5).** `runFFmpeg` discards termination status; `stop()` unconditionally deletes `.mic.wav`/`.system.wav` after the merge call, and `recoverOrphan` deletes them BEFORE its output-existence check; the no-ffmpeg fallback keeps mic-only and still deletes `system.wav` (`MeetingRecorder.swift:914-917,1046-1051,1065-1068,1124-1128`).
