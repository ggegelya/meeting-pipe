## How to pick up a task (delegation)

Mechanics are codified in `/tech-task <ID>` (read the task here, read the orientation docs, implement, verify, one commit on `main`, do not push). Soft context that command does not carry:

- Task ids carry no `TECH-` prefix (e.g. `/tech-task START1`). Historical code comments, ADRs, and commits keep the old `TECH-` form as provenance.
- **Full specs for items carried from Q4 unchanged live in [`q4-final.md`](./q4-final.md)** (search the id); this file restates them in one line and only writes full specs for new or changed items, so the spec has a single owner.
- User locale: 99% English / Ukrainian. Do not default examples to German. Verify any new `uk` labels in `MuteLabels.toml` with the user.
- The user is vibe-coding: high-level explanations, does not read code. Lean on the ARCHITECTURE.md diagrams. Blind sub-pixel visual churn is out of bounds; visual changes need eyes (the binding constraint that deferred the DSN3 font sweep and DSN9 retune).
- No em-dashes in any output (CI fails on a U+2014 in an added line under `daemon/Sources` and `daemon/Resources`). Use hyphens, commas, or rewrite. Commit with the repo's git identity. Do not push without permission.
- Binding constraint (CLAUDE.md): the primary user is the author; selling is tertiary; technical excellence in core functionality comes first. The BRAND launch band stays deferred. The one standing promotion: GitHub repo presence + visual identity (closed in Q4 as REPO1/REPO2).
- New families this quarter: `AI*` (on-device intelligence) and `DV*` (the meeting-level data view). Every AI/DV capability must route through the local engine path so the `egress_guard` clamps it under regulated/NDA, honour `effective_backend()` (forced local under regulated/NDA), and add no new always-on egress.

## Task specs

### Intelligence layer (new, AI band)

Origin: two deep-research reports ([local AI intelligence](../research/2026-06-26-local-ai-intelligence.md), [meeting-level UX](../research/2026-06-26-meeting-level-ux.md)). The competitor scan (Granola, Notion AI, Otter, Fathom, Fireflies, tl;dv, Reflect, Mem, Hyprnote) found that ask-across-all chat is now table stakes but the genuinely under-served seams are action-item lifecycle (capture-but-never-tracked) and scheduled digests; trend dashboards and speaker analytics are team-shaped and skipped for a single user. Hyprnote is the posture twin (fully on-device, BYO-LLM) and shares meeting-pipe's exact gaps, which is the differentiation opening. The load-bearing feasibility risk is long-context generation latency: MLX prefills the full context before the first token, so the lowest-risk first builds are latency-tolerant (the lifecycle is LLM-free; the digest runs in the background), and anything interactive (chat) is gated behind a measured spike. Every AI item routes through the local engine path so `egress_guard` clamps it under regulated/NDA and honours `effective_backend()`, adding no new always-on egress.

**[DONE] AI1 (P1): action-item lifecycle, resolved/done flag + aging.** The highest-leverage, lowest-risk item and the named next step in `pipeline/src/mp/actions.py`; both research tracks ranked it first. Mirror the schema change in the Swift `MeetingMetaSidecar` contract and every publisher (touch both sides + `test_workflow_overlay.py`). Acceptance: `mp actions --open/--closed/--overdue` works with aging on dated actions; the flag round-trips through the sidecar across a republish (upsert, no duplicate); pipeline pytest + ruff and the daemon build are green. Continues FEAT4. No UI here (that is DV1/AI2's surface plus the close control in DV1). Done: `ActionItem.resolved` (pydantic `bool`, default `False`, `validation_alias` accepting the legacy `done` spelling, serialized as `resolved`) is mirrored on the Swift `MeetingSummary.ActionItem` decoder + `jsonObject()` write-bridge, both tolerant of an absent flag (legacy items decode as open). One correction to the spec's wording: action items live in `<stem>.summary.json`, not the `MeetingMetaSidecar` (`meta.json`) workflow sidecar, so the "two decoders" are `schemas.py` and `MeetingSummary.swift`, and the round-trip works because `publish_router.fanout` re-reads `summary.json` on republish (it never regenerates it). `mp actions` gained the mutually-exclusive `--open`/`--closed`/`--overdue` filters and a stdlib-`date` age (day count off `due`, "Nd overdue"/"in Nd"/"due today") shown for dated open actions, plus `resolved`/`age_days`/`overdue` in `--json`. **[DONE] AI2 (P1, spike): on-device embedding index + long-context RAG latency and faithfulness.** Gates everything chat-shaped. Done: shipped `embed_index.py` (pluggable embedder: `MLXEmbedder` multilingual-e5-small on MLX for production, numpy `HashingEmbedder` for CI; chunked via `chunked_windows`; cosine search; the durable artifact AI3 imports) and `mp ai2-spike` (streaming TTFT capture, needle-in-haystack faithfulness/citation, per-size incremental results, go/no-go recommender). Prefill throughput falls with context (the predicted MLX collapse). Verdict: NO-GO for interactive live chat, so AI3 is async, 14B for citations, context budget ~4K (16K unsafe on this Mac); also found the embedding model must load with `local_files_only` / `HF_HUB_OFFLINE` so it does not trip the egress guard under regulated/NDA. Origin: the [meeting-level UX report](../research/2026-06-26-meeting-level-ux.md). **[DONE] DV2 (P2, spike): attendee quality for a People pivot.** Before building a People view, inspect real `summary.json` `attendees[]` across the corpus and measure the cross-meeting dedup hit-rate (how often the same person is named consistently). Acceptance: a written read on attendee data quality and dedup rate, and a decision on whether a People pivot is trustworthy now or must wait for FEAT3-ROSTER (voiceprint identity) / CAL1 (calendar attendee seeding). No People view ships until this says the data supports it. Done: shipped a re-runnable stdlib audit (`docs/spikes/dv2_attendee_audit.py`) that classifies each `attendees[]` value as a real name vs a diarization artifact (`speaker_*` / bare index) and measures fill rate, artifact ratio, distinct names, cross-meeting recurrence, and the dedup hit-rate, plus the decision doc (`docs/spikes/dv2-attendee-quality.md`). Verdict: NO-GO, confirmed on the owner's real 165-meeting library (transferred as a privacy-safe aggregate, `docs/spikes/dv2-attendee-audit-results.json`; raw names never left that machine). 878 attendee mentions: 65% are diarization artifacts (`speaker_unknown`, bare cluster indices, `speaker_N`) and only 35% are candidate real names (189 distinct, 34 recurring across >=2 meetings), for an 18% cross-meeting dedup rate. Against the proposed bar (>=15-20 summaries, artifact_ratio < ~0.2, dedup_hit_rate >= ~0.3) the corpus size passes but the two quality gates fail (0.65 and 0.18). The load-bearing finding is structural, not sample-size: `attendees[]` is model-inferred from FluidAudio `speaker_*` labels with no identity signal for un-named speakers pre-FEAT3, which the real data confirms is the dominant blocker; the field cannot carry a trustworthy cross-meeting roster until voiceprint identity (FEAT3-VOICEPRINT -> FEAT3-ROSTER) turns `speaker_N` into names or calendar seeding (CAL1) supplies clean ones. A genuine recurring-people signal does exist (34 people), so this is "not yet," not "never"; the 18% is also an exact-match floor (fuzzy-dedup of name variants like "Sarah"/"Sarah Chen" would lift it, but not past the 65% artifact gap). A People view stays deferred; DV1 (Facts, identity-free) is the DV theme's shipped value.
