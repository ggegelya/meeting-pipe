## How to pick up a task (delegation)

Mechanics are codified in `/tech-task <ID>` (read the task here, read the orientation docs, implement, verify, one commit on `main`, do not push). Soft context that command does not carry:

- Task ids carry no `TECH-` prefix (e.g. `/tech-task START1`). Historical code comments, ADRs, and commits keep the old `TECH-` form as provenance.
- **Full specs for items carried from Q4 unchanged live in [`q4-final.md`](./q4-final.md)** (search the id); this file restates them in one line and only writes full specs for new or changed items, so the spec has a single owner.
- User locale: 99% English / Ukrainian. Do not default examples to German. Verify any new `uk` labels in `MuteLabels.toml` with the user.
- The user is vibe-coding: high-level explanations, does not read code. Lean on the ARCHITECTURE.md diagrams. Blind sub-pixel visual churn is out of bounds; visual changes need eyes (the binding constraint that deferred the DSN3 font sweep and DSN9 retune).
- No em-dashes in any output (CI fails on a U+2014 in an added line under `daemon/Sources` and `daemon/Resources`). Use hyphens, commas, or rewrite. Commit with the repo's git identity. Do not push without permission.
- Binding constraint (CLAUDE.md): the primary user is the author; selling is tertiary; technical excellence in core functionality comes first. The BRAND launch band stays deferred. The one standing promotion: GitHub repo presence + visual identity (closed in Q4 as REPO1/REPO2).
- New families this quarter: `AI*` (on-device intelligence) and `DV*` (the meeting-level data view). Every AI/DV capability must route through the local engine path so the `egress_guard` clamps it under regulated/NDA, honour `effective_backend()` (forced local under regulated/NDA), and add no new always-on egress.

- **P2**: polish and power-user payoff.

### Intelligence layer (new, AI band)

Origin: two deep-research reports ([local AI intelligence](../research/2026-06-26-local-ai-intelligence.md), [meeting-level UX](../research/2026-06-26-meeting-level-ux.md)). The competitor scan (Granola, Notion AI, Otter, Fathom, Fireflies, tl;dv, Reflect, Mem, Hyprnote) found that ask-across-all chat is now table stakes but the genuinely under-served seams are action-item lifecycle (capture-but-never-tracked) and scheduled digests; trend dashboards and speaker analytics are team-shaped and skipped for a single user. Hyprnote is the posture twin (fully on-device, BYO-LLM) and shares meeting-pipe's exact gaps, which is the differentiation opening. The load-bearing feasibility risk is long-context generation latency: MLX prefills the full context before the first token, so the lowest-risk first builds are latency-tolerant (the lifecycle is LLM-free; the digest runs in the background), and anything interactive (chat) is gated behind a measured spike. Every AI item routes through the local engine path so `egress_guard` clamps it under regulated/NDA and honours `effective_backend()`, adding no new always-on egress.

**[DONE] AI1 (P1): action-item lifecycle, resolved/done flag + aging.** The highest-leverage, lowest-risk item and the named next step in `pipeline/src/mp/actions.py`; both research tracks ranked it first. Add an optional `resolved`/`done` state to `ActionItem` (`pipeline/src/mp/schemas.py`) tolerant on both decoders, compute aging off the existing ISO `due` date, and teach `mp actions` `--open` / `--closed` / `--overdue` with an age for dated open actions. Mirror the schema change in the Swift `MeetingMetaSidecar` contract and every publisher (touch both sides + `test_workflow_overlay.py`). Acceptance: `mp actions --open/--closed/--overdue` works with aging on dated actions; the flag round-trips through the sidecar across a republish (upsert, no duplicate); pipeline pytest + ruff and the daemon build are green. Continues FEAT4. No UI here (that is DV1/AI2's surface plus the close control in DV1). Done: `ActionItem.resolved` (pydantic `bool`, default `False`, `validation_alias` accepting the legacy `done` spelling, serialized as `resolved`) is mirrored on the Swift `MeetingSummary.ActionItem` decoder + `jsonObject()` write-bridge, both tolerant of an absent flag (legacy items decode as open). One correction to the spec's wording: action items live in `<stem>.summary.json`, not the `MeetingMetaSidecar` (`meta.json`) workflow sidecar, so the "two decoders" are `schemas.py` and `MeetingSummary.swift`, and the round-trip works because `publish_router.fanout` re-reads `summary.json` on republish (it never regenerates it). `mp actions` gained the mutually-exclusive `--open`/`--closed`/`--overdue` filters and a stdlib-`date` age (day count off `due`, "Nd overdue"/"in Nd"/"due today") shown for dated open actions, plus `resolved`/`age_days`/`overdue` in `--json`. Publishers reflect the flag: Notion `to_do.checked`, the `[x]`/`[ ]` checkbox in `summarize._render_summary_md` / `publish_fs` / `publish_obsidian`, the BYO-paste `[x]` parse, and `actions.json` via `model_dump`. The daemon Edit-Save threads `resolved` through `EditableAction` so a text correction never reopens a closed action. Tests: `test_schemas`, `test_actions` (filters + aging + alias), `test_publish_fs` / `test_publish_obsidian` / `test_publish_notion_blocks` (checkbox/checked), `test_publish_router` (republish upsert round-trip), `test_publish_from_paste` (markdown round-trip), a `test_workflow_overlay` guard that the meta-overlay does not shadow per-action state, and Swift `MeetingSummaryTests` (decode both spellings + `jsonObject` round-trip). pytest 323 passing (the 14 pre-existing secrets-env failures unchanged), ruff clean, daemon debug build + `swift test` (849) green.

**AI2 (P1, spike): on-device embedding index + long-context RAG latency and faithfulness.** Gates everything chat-shaped. Build an on-device embedding index over the library (reusing the local engine; embeddings are cheap, ~tens of K tok/s on M-series) and a small vector index, then measure RAG-synthesis time-to-first-token and total wall-clock at 4K / 8K / 16K context on the user's actual Mac, plus a faithfulness/citation-accuracy read on multi-document answers. The cited 6.9s figure is a short-prompt number; MLX full-prefill collapses throughput at longer context, which is exactly RAG's worst case. Acceptance: an index is built; measured TTFT + wall-clock at the three context sizes and a citation-accuracy read are recorded on the real hardware; the spike returns a go/no-go on interactive chat vs async-only and a recommended context budget. Folds FEAT2's deferred vector-RAG follow-up. Depends on nothing; gates AI3.

**AI3 (P2): Ask-AI, engine-backed cited answers + a daemon surface.** Continues FEAT2 (the CLI-only `mp ask` TF-IDF MVP). Replace the ranker with retrieval-augmented answers that carry at least one verifiable `<stem>` citation, route through `httpx` so the egress guard clamps it, and use the forced-local engine with no cloud fallback under regulated/NDA. Surface it in the daemon (a QuickFind-style "ask my meetings" entry, quiet register, no sparkle). Scope the UX (interactive vs async) only after AI2. Acceptance: a natural-language answer with a real citation, within AI2's latency budget, clamped under regulated/NDA; pytest + ruff green; the daemon surface is reachable. Depends on AI2.

**AI4 (P2): weekly review digest, async and on-device.** A scheduled, locally generated digest of the week's aging open actions and recent decisions, written to disk and publishable through the existing fanout. The single-user-relevant slice of "trend analysis", and latency-tolerant because it runs in the background. Acceptance: a digest is generated on-device from decisions + open actions (a first version needs no RAG), routed through `httpx` so the egress guard clamps it under regulated/NDA and honours `effective_backend()`; no new always-on egress. Depends on AI1.

**AI5 (P3, spike): meeting-type classification.** Prototype on-device meeting-type labels over the existing library (a short local call, or a heuristic over attendees / cadence / title) and show the results to the owner before committing a permanent `MeetingSummary` field. Uncommon among competitors (only MeetGeek auto-detects). Acceptance: a throwaway classifier labels the library by type; the spike returns a recommendation on whether the labels are useful enough to wire into grouping/digests and earn a schema field; no schema change lands until that decision.

### Meeting-level data view (new, DV band)

Origin: the [meeting-level UX report](../research/2026-06-26-meeting-level-ux.md). Verdict: meeting-pipe earns a meeting/facts-altitude view, but as a derived, read-only, zero-egress projection built from facts already on disk (ADR 0003: the filesystem is the source of truth; a data view must be an index over files, not a new database of record), not a CRM. The competitor scan (Granola People/Companies, Obsidian Bases `attendees.contains(this)`, Otter My Action Items, Notion saved views) converges on lightweight derived entity pages + an action inbox; the heavy coaching/scorecard end is out of register. The Library today is date/status/workflow-only (`LibraryScope`) with no person/decision/topic pivot.

**DV1 (P1): cross-meeting facts data-view.** The "data view" the owner sensed was missing, and FEAT4's missing UI half. A new facts-altitude view reachable from the Library rail (a new `LibraryScope`) that aggregates `decisions[]` and open `actions[]` (post-AI1, with their resolved state) across the whole library read-only, each row linking back to its meeting `<stem>`, plus a control to mark an action done (persisted through the sidecar so it survives a republish). Quiet register, native idioms (a `Table` or grouped list, not a dashboard). Acceptance: the view lists open actions and recent decisions across meetings, each navigates to its source meeting, an action can be marked done from here and the state round-trips through AI1's sidecar; no a11y or token regressions; daemon builds debug + release. Depends on AI1.

**DV2 (P2, spike): attendee quality for a People pivot.** Before building a People view, inspect real `summary.json` `attendees[]` across the corpus and measure the cross-meeting dedup hit-rate (how often the same person is named consistently). Acceptance: a written read on attendee data quality and dedup rate, and a decision on whether a People pivot is trustworthy now or must wait for FEAT3-ROSTER (voiceprint identity) / CAL1 (calendar attendee seeding). No People view ships until this says the data supports it.
